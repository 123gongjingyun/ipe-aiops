## 📊 **Redis配置架构总览**

### 1. **Redis单节点架构** (44-56行)
- 12个配置规格
- 适用于开发测试、小型应用场景
- 无高可用保障

### 2. **Redis 3主3从集群架构** (91-100行) ⭐ 新增
- 8个配置规格
- **架构特点**：
  - 3个主节点，每个主节点1个从节点（共6个节点）
  - 支持数据自动分片，提供3个分片
  - 具备高可用性，支持故障自动转移
  - 性能约为单节点的3-4倍

### 3. **Redis一主2从3哨兵架构** (101-110行) ⭐ 新增
- 8个配置规格  
- **架构特点**：
  - 1个主节点，2个从节点，3个哨兵节点（共6个节点）
  - 哨兵负责监控和自动故障转移
  - 支持读写分离，主节点写入，从节点读取
  - 性能约为单节点的1.5-2倍

## 🔍 **集群架构详细对比**

### **3主3从集群架构**
```
适用场景：大型企业应用，需要数据分片和高可用
核心优势：
• 数据分片：支持3个分片，数据均匀分布
• 高可用性：每个分片都有主从，支持故障自动转移  
• 水平扩展：可以根据业务需要增加主从节点数量
• 读写性能：集群总性能约为单节点的3-4倍
• 内存容量：集群总内存约为单个节点的3倍

配置规格范围：
• 小型集群：2C4G x 6节点，支持日活<1万
• 中型集群：4C8G x 6节点，支持日活<20万  
• 大型集群：8C16G x 6节点，支持日活<200万
• 超大型集群：16C32G x 6节点，支持日活>500万
```

### **一主2从3哨兵架构**
```
适用场景：需要高可用但不需要数据分片的应用
核心优势：
• 高可用性：哨兵自动监控和故障转移
• 读写分离：主节点写入，从节点分担读压力
• 简单架构：相对集群架构更容易运维
• 事务支持：支持Redis事务和Lua脚本
• 性能提升：读性能提升约1.5-2倍

配置规格范围：
• 小型高可用：2C4G主 + 2C4G从x2 + 1C2G哨兵x3，日活<5000
• 中型高可用：4C8G主 + 4C8G从x2 + 1C2G哨兵x3，日活<10万
• 大型高可用：8C16G主 + 8C16G从x2 + 2C4G哨兵x3，日活<200万
• 超大型高可用：16C32G主 + 16C32G从x2 + 2C4G哨兵x3，日活>500万
```

## 💾 **存储成本对比**

以4C8G配置为例：

### **3主3从集群架构**
- **总资源需求**: 4C8G x 6节点 = 24核48G内存
- **存储成本**: 基准成本 × 6
- **适用规模**: 日活<20万
- **成本效益比**: 适合需要数据分片的大规模应用

### **一主2从3哨兵架构** 
- **总资源需求**: 4C8G主 + 4C8G从x2 + 1C2G哨兵x3 = 11核22G内存
- **存储成本**: 基准成本 × 2.75
- **适用规模**: 日活<10万  
- **成本效益比**: 适合需要高可用但不需要分片的应用

## 🎯 **选择建议**

### **选择3主3从集群架构的场景**
✅ 数据量大，需要数据分片  
✅ 写入频繁，需要分布式的写入能力  
✅ 预算充足，可以承担6节点的成本  
✅ 需要水平扩展能力  

### **选择一主2从3哨兵架构的场景**  
✅ 需要高可用但不需数据分片  
✅ 读多写少，可以充分利用读写分离  
✅ 预算有限，希望控制成本  
✅ 需要支持Redis事务和Lua脚本  

### **选择单节点架构的场景**
✅ 开发测试环境  
✅ 小型应用，数据量不大  
✅ 预算非常有限  
✅ 可以接受短暂的服务中断

每个架构类型都提供了从开发测试到超大规模生产的完整配置梯度：
1. **小型**: 日活<1万（集群）/ <5000（哨兵）

### **性能指标设计**
- **集群总连接数**: 考虑了所有节点的连接数总和
- **集群总QPS**: 考虑了读写分离和数据分片的总性能
- **内存使用率**: 控制在80%以下，确保系统稳定性
- **分片数量**: 3主3从架构固定为3个分片
