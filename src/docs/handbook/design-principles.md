# 设计原则

平台设计思维框架与编码质量标准。

---

## 设计思维模式

### 1. 角色分离

始终明确两个视角：应用担当（申请/验收）vs 基础担当（交付/交付确认）。每个功能的入口、信息密度和操作权限都围绕角色定义，避免角色混用导致界面复杂度膨胀。

### 2. 配置层与运行时分离

服务目录定义"能提供什么"（Spec），工单跟踪"交付了什么"（Order/Instance）。配置层变更不影响运行时数据，运行时数据反向驱动配置层优化。两层通过 `sourceSpecId` 关联。

### 3. AI 编排、人类审核

AI 生成推荐方案（资源编排、参数建议、流程选择），用户确认/调整后执行。AI 是加速器而非替代者，关键决策节点必须保留人工确认。

### 4. 渐进披露

默认展示摘要信息（卡片、列表、状态标签），按需展开详情（抽屉、弹窗、折叠面板）。避免信息过载，让用户在正确的时间看到正确粒度的信息。

### 5. 参考数据驱动

新增服务必须参考 `refs/原子服务定义.md` 中的标准分类和字段定义，不随意发明新的服务类型或字段。所有服务数据来源于 BOM 配置，不硬编码。

## 代码质量原则

| 原则 | 说明 |
|------|------|
| 单一职责 (SRP) | 每个函数、类、模块、组件应只做一件事，保持职责清晰 |
| DRY | 避免重复代码，提取共用逻辑到单独的函数、类、模块或可复用单元 |
| KISS | 保持代码简洁明了，避免过度设计 |
| YAGNI | 不要实现当前不需要的功能，避免过度抽象 |
| 模块化 | 按功能/领域划分模块，保持关注点分离 |
| 可测试性 | 编写可测试的代码，避免紧耦合，便于单元测试和集成测试 |
| 可维护性 | 代码结构清晰，便于理解和修改 |

## 命名规范

### 配置层

格式：`[Qualifier]Service[Aspect]Spec`，必须包含 `Service` 前缀或中缀。

示例：`AtomicServiceSpec`、`ServiceAssemblySpec`、`ComboServiceSpec`、`ServiceSLASpec`、`ServiceFlowSpec`

### 运行时层

按领域选用最合适的术语，不强制 Spec 后缀。

示例：`ServiceOrder`、`AtomicServiceInstance`、`ServiceMetric`、`ServiceFlowRecord`

## 编码规范

| 规范 | 说明 |
|------|------|
| 命名 | 使用描述性命名，遵循项目命名约定，避免不必要的缩写 |
| 风格一致 | 遵循项目或语言的官方风格指南，使用指定的格式化和静态检查工具 |
| 注释文档 | 为复杂逻辑、算法、业务规则添加注释，优先通过清晰的命名使代码自解释 |
| 测试标识 | 前端关键元素添加 `data-test-id` 属性，后端关键函数/类添加测试友好的设计 |

## 错误处理

| 原则 | 说明 |
|------|------|
| 异常处理 | 正确处理边缘情况和错误，提供有用的错误信息 |
| 错误边界 | 在关键位置设置错误边界（前端错误边界、后端中间件），防止错误扩散 |
| 用户反馈 | 向用户提供清晰的错误提示和操作建议 |
| 日志记录 | 记录关键操作和错误日志，便于问题排查和监控 |
