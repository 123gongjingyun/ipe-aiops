# 服务交付门户架构方案

> **文档性质**：对外讨论版  
> **讨论主题**：服务交付门户与现有ITSM系统的边界划分、集成策略、服务目录标准化  
> **编制日期**：2026年6月3日

---

## 一、背景与问题定义

### 1.1 现状

当前所有IT相关申请（资源交付、配置变更、应用发布、故障工单、权限申请等）均混合在现有 **ITSM工单系统** 中统一流转。随着交付运营复杂度增加，计划将 **IT资源交付、配置变更、应用发布等交付运营类服务** 抽离出来，构建独立的 **服务交付门户**（Service Delivery Portal）。

### 1.2 核心问题

抽离后面临两个关键架构决策：

## 二、方案对比分析

### 2.1 方案A：交付门户完全独立（闭环模式）

**架构描述**：
- 交付门户自建完整的流程引擎、表单引擎、审批引擎、权限体系
- 与ITSM系统完全解耦，独立运行
- 所有交付类申请的端到端流程在交付门户内闭环完成

```
┌─────────────────────────────────────────────────────────────────────────────┐

│   │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────────┐ │    │
│   │  │ 服务目录  │  │ 申请表单  │  │ 审批引擎  │  │ 交付执行 + 状态   │ │    │
│   │  │          │→ │          │→ │          │→ │ 跟踪             │ │    │

│   └──────────────┘                                                        │
│                                                                             │
│   ┌──────────────────────────────────────────────────────────────────┐    │
│   │                     ITSM系统（现有）                              │    │
│   │  故障工单 / 权限申请 / 非交付类流程  ←── 继续独立运行              │    │
│   └──────────────────────────────────────────────────────────────────┘    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

**优势**：

| 优势 | 说明 |
|------|------|
| **用户体验一致** | 申请、审批、跟踪、交付全在一个门户完成，无需跳转 |
| **开发灵活性高** | 不受ITSM系统接口限制，可快速迭代交付特有的功能（如AI配置推荐、网络链路可视化） |
| **数据统一** | 服务目录、配置档案、交付进度、资源状态全部在一个系统内，数据分析更简单 |
| **响应速度快** | 无需跨系统接口调用，审批和交付响应更快 |
| **无单点依赖** | ITSM系统故障不影响交付门户运行 |

**劣势**：

| 劣势 | 说明 |
|------|------|
| **重复建设** | 与ITSM系统在审批流、权限、通知等能力上存在重复建设 |
| **流程割裂** | 交付类流程与非交付类流程（如故障工单、权限申请）分离，用户需要在两个系统间切换 |
| **审计合规** | 若公司要求所有变更必须走ITSM统一审计，闭环模式可能需要额外的数据同步机制 |
| **数据孤岛** | 交付门户的工单数据与ITSM不互通，管理层看全局报表时需要拉通两个系统 |
| **推广成本** | 需要说服业务部门接受新的审批入口，改变原有ITSM使用习惯 |

**适用场景**：

- 组织内ITSM系统老化，扩展性差
- 交付运营团队有独立的技术能力和运维权限
- 组织对交付效率要求极高，无法容忍跨系统延迟
- 交付类流程占比高（>50%），值得独立建设

### 2.2 方案B：交付门户与ITSM对接（集成模式）

**架构描述**：
- 交付门户作为**前端界面层**，负责服务目录展示、申请表单渲染、交付状态跟踪
- **审批流在ITSM系统内流转**，利用ITSM成熟的审批引擎
- 用户提交申请后，交付门户自动调用ITSM接口创建工单
- ITSM审批完成后，通过回调/Webhook通知交付门户
- 交付门户根据审批结果执行资源交付，并将交付状态回写ITSM

**优势**：

| 优势 | 说明 |
|------|------|
| **复用投资** | 充分利用ITSM已有的审批引擎、流程引擎、权限体系、审计日志，避免重复建设 |
| **流程统一** | 所有变更（交付类 + 非交付类）的审批流都在ITSM内统一管理，审计合规 |
| **数据贯通** | ITSM作为"单据中枢"，管理层可在ITSM看全局报表，交付门户作为专业前端 |
| **推广成本低** | 审批人无需改变ITSM使用习惯，仍在熟悉的界面审批 |
| **风险可控** | 不改变现有ITSM核心流程，交付门户作为增量系统接入 |

**劣势**：

| 劣势 | 说明 |
|------|------|
| **用户体验割裂** | 用户在交付门户提交申请，审批人需要在ITSM审批，用户需要跨系统查看审批进度 |
| **接口依赖重** | 高度依赖ITSM的API能力和稳定性，接口变更可能导致交付门户故障 |
| **响应延迟** | 跨系统调用增加延迟，审批状态同步可能存在秒级/分钟级延迟 |
| **功能受限** | 交付门户的AI推荐、配置档案、网络可视化等特色功能，需要以ITSM工单字段为载体传递，可能受限于ITSM字段扩展能力 |
| **联调复杂** | 需要与ITSM团队深度协作，接口联调、异常处理、幂等设计等工作量大 |
| **单点风险** | ITSM系统故障时，交付门户的审批流也会中断 |

**适用场景**：
- ITSM系统功能完善，扩展性良好，API开放
- 组织内有强审计合规要求，所有变更必须走ITSM
- 交付类流程占比较低，不值得独立建设审批引擎
- ITSM团队支持配合接口开发和维护

### 2.3 方案C：混合架构（推荐方案）

**架构描述**：
- **前台独立**：交付门户自建轻量级审批流，用于**简单申请**（如开发环境资源、测试环境搭建）的快速审批和自动审批
- **复杂申请联动ITSM**：对于**生产环境变更、高SLA服务申请**等复杂场景，自动推送到ITSM走正式审批流程
- **状态双向同步**：无论哪种模式，交付状态和审批状态都实时同步到两个系统

**分类规则示例**：

| 申请类型 | 环境 | SLA | 推荐审批模式 | 理由 |

| UAT测试环境搭建 | UAT | 银牌 | **门户内轻量审批** | 风险低，追求效率 |
| 生产环境资源申请 | PROD | 金牌 | **推送ITSM正式审批** | 风险高，需严格审计 |
| 网络架构变更（VPC/防火墙） | 任意 | 任意 | **推送ITSM正式审批** | 影响面大，需多方评审 |
| 应用发布 | PROD | 金牌 | **推送ITSM正式审批** | 需与变更管理联动 |
| 配置参数优化 | UAT | 银牌 | **门户内轻量审批** | 风险可控 |

**优势**：

| 优势 | 说明 |
|------|------|
| **兼顾效率与合规** | 简单申请秒级完成，复杂申请严格审计 |
| **渐进式迁移** | 初期大部分申请走ITSM，随着门户成熟度提升，逐步将低风险申请转为门户内闭环 |
| **用户体验优化** | 申请人始终在交付门户操作，审批人根据申请复杂度在不同系统审批 |
| **风险可控** | 生产环境变更仍走ITSM，不触碰审计红线 |
| **复用与独立平衡** | 既复用了ITSM的成熟能力，又保留了交付门户的独立性 |

**劣势**：

| 劣势 | 说明 |
|------|------|
| **架构复杂度** | 需要同时维护两套审批模式，逻辑分支增加 |
| **分类规则维护** | 需要定义和维护"简单申请"vs"复杂申请"的分类规则 |
| **数据一致性** | 门户内闭环的工单数据与ITSM不互通，需要定期对账 |

| **交付效率** | ⭐⭐⭐ 最高 | ⭐⭐ 受接口延迟影响 | ⭐⭐⭐ 简单申请秒级，复杂申请合规 |

| **单点故障风险** | 低 | 高（依赖ITSM） | 中 |
| **未来扩展性** | ⭐⭐⭐ 灵活 | 受ITSM限制 | ⭐⭐⭐ 灵活 |

## 四、推荐方案与实施建议

### 4.1 推荐方案：方案C（混合架构）

**理由**：
1. **渐进式演进**：不需要一次性推翻ITSM流程，而是从低风险申请开始逐步迁移
2. **业务价值最大化**：简单申请（占日常70%以上）在门户内秒级完成，显著提升交付效率
3. **风险可控**：生产环境等高风险变更仍走ITSM，不触碰审计红线
4. **投资保护**：充分利用ITSM已有的审批引擎和权限体系，不重复建设

### 4.2 实施路线图

```
Phase 1（本期）：基础能力建设
├── 交付门户：服务目录 + 配置数据字典 + 申请表单 + 交付跟踪
├── 审批模式：全部申请推送ITSM（集成模式）
└── 目标：验证门户价值，用户习惯培养

Phase 2（3个月后）：轻量审批试点
├── 审批模式：DEV/UAT环境申请转为门户内轻量审批
├── 生产环境申请仍推送ITSM
└── 目标：提升低风险申请效率

Phase 3（6个月后）：全面混合
├── 审批模式：按申请类型和环境自动路由审批系统
├── 完善分类规则和自动审批策略
└── 目标：效率与合规的最佳平衡
```

### 4.3 与ITSM的接口设计（方案C + Phase 1）

**交付门户 → ITSM（单向推送）**：

interface CreateITSMWorkOrderRequest {

- 交付执行过程中 → 定期回写执行进度到ITSM（如30%、60%、90%）
- 交付完成 → 回写完成状态到ITSM

### 5.3 建议：保持现有命名，增加标准化映射

**理由**：

| 维度 | 分析 |
|------|------|
| **现有命名优势** | 当前命名（云服务器开通/MySQL部署/FW策略等）是技术人员熟悉的实操术语，直观易懂 |
| **标准命名劣势** | "计算服务""平台服务"等标准术语过于抽象，用户难以快速定位到具体操作 |
| **用户习惯** | 用户习惯说"我要开通一台MySQL"而不是"我要申请数据库服务" |
| **交付效率** | 越具体的服务名称，越能减少用户的选择成本 |

**建议方案**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    服务目录命名策略（推荐）                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  面向用户的展示层：保持现有命名（直观、具体、技术人员熟悉）                    │
│                                                                             │
│  示例：                                                                      │
│  • "云数据库开通"（而非"数据库服务"）                                       │
│  • "MySQL部署"（而非"数据库实例创建"）                                      │
│  • "FW策略开通"（而非"防火墙规则服务"）                                     │
│                                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  面向ITSM的标准化映射层：每个服务绑定标准分类标签（用于报表和审计）             │
│                                                                             │
│  示例：                                                                      │
│  • 云数据库开通 → ITSM分类：平台服务 > 数据库服务                           │
│  • MySQL部署    → ITSM分类：平台服务 > 数据库服务                           │
│  • FW策略开通   → ITSM分类：基础设施服务 > 网络安全服务                       │
│  • 容器集群配置 → ITSM分类：平台服务 > 容器云服务                           │
│                                                                             │
│  实现方式：每个服务规格增加 `itsmCategory` 字段                              │
│                                                                             │
│  ```typescript                                                              │
│  interface AtomicServiceSpec {                                              │
│    // ... 现有字段                                                          │
│    itsmCategory?: {          // 新增：ITSM标准分类映射                       │
│      level1: string;         // 一级分类：基础设施服务/平台服务/安全服务      │
│      level2: string;         // 二级分类：数据库服务/中间件服务/网络安全服务  │
│    };                                                                       │
│  }                                                                          │
│  ```                                                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**这样做的好处**：
1. **用户体验不变**：技术人员看到熟悉的术语，快速找到所需服务
2. **审计合规满足**：ITSM报表可按标准分类统计，满足管理层视角
3. **未来扩展灵活**：如需对外提供标准化服务目录，可通过映射层转换
