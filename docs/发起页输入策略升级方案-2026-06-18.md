# IPE/AIOps 发起页输入策略升级方案

> 创建日期：2026-06-18  
> 适用范围：`project-mywayaiops-refactor-V1` Portal 发起页 / Center 服务目录与设置页 / Shared 数据模型  
> 目标：替代客户线下 Excel 需求模板的首轮沟通成本，收敛为“业务描述 + 架构图 + AI 引导补全”的线上发起模式

---

## 一、背景与问题判断

当前客户侧存在一类真实问题：

- 不同需求往往对应不同的 Excel 申请模板
- 模板字段多、解释弱，用户常常不知道该填什么
- 模板填写后仍需 2-3 轮沟通，才能形成可评审的申请材料
- 真正对资源边界更有价值的信息，通常不是模板本身，而是业务目标和现状/目标架构图

这说明，Excel 模板本质上承担了 3 类本应由系统完成的职责：

1. 需求引导：告诉用户要补哪些信息
2. 结构化提取：把自然语言和附件转成可评审字段
3. 合理性校验：判断配置是否缺失、冲突或不符合目标环境

如果继续把“下载模板 -> 线下填写 -> 上传模板”作为主入口，只是把线下沟通搬到线上，不会真正降低用户成本。

因此，发起页的主设计应从“模板驱动”切换为“AI 引导驱动”。

---

## 二、核心结论

### 2.1 主路径选择

发起页建议采用以下主路径：

`业务描述 + 架构图 -> AI 解析 -> 生成需求细化稿 / 申请草稿 -> 用户确认 -> 导出材料`

而不是：

`先下载 Excel 模板 -> 用户自行填写 -> 上传后再人工澄清`

### 2.2 对“配置文档模板”的重新定位

“配置文档模板”不建议作为发起页的核心入口，而应降级为以下 3 类辅助能力：

1. 导入能力
   - 兼容客户已有 Excel 需求模板
   - 用于把历史材料导入系统，减少迁移阻力

2. 导出能力
   - 用户在线完成申请后，导出为 Excel / PDF
   - 用于评审、审批、归档、跨团队传递

3. 参考能力
   - 作为页面内的“示例模板 / 填写说明”
   - 帮助用户理解典型场景需要哪些字段

结论：保留模板，但不再让模板主导首轮交互。

### 2.3 架构图的优先级

建议把“上传架构图”提升为发起页的推荐主入口，原因如下：

- 架构图更接近实际环境需求，而不是抽象字段
- 架构图天然包含组件、环境、依赖、网络、边界、冗余关系
- 用户往往不会描述“需要几节点、什么主从、什么备份等级”，但能给出现网图或目标图
- AI 可以据此先生成一版候选配置，再让用户做补充确认

## 三、输入模式设计

发起页建议支持三种输入模式并存，但引导顺序要有主次：

### 3.1 模式 A：架构图优先模式（推荐主路径）

用户输入：

- 业务目标 / 场景说明
- 上传架构图
- 可补充少量关键上下文（如目标环境、上线时间、关键系统）

AI 输出：

- 识别出的组件清单
- 推断的环境层次与依赖关系
- 推荐的资源/中间件/网络/安全申请项
- 缺失信息清单
- 初步合理性建议

适用场景：

- 用户已有现状架构图或目标架构图
- 用户不清楚具体字段，但清楚整体建设目标

### 3.2 模式 B：业务描述模式

用户输入：

- 纯自然语言描述
- 可选填写业务名称、环境、优先级、目标时间

AI 输出：

- 一版标准化需求草稿
- 推荐的服务组合与原子服务
- 需要继续补充的关键字段

适用场景：

- 还没有架构图
- 需求处于较早阶段

### 3.3 模式 C：配置文档导入模式（辅助兼容）

用户输入：

- 上传客户已有 Excel 模板
- 可选上传架构图

AI 输出：

- 自动抽取模板中的结构化字段
- 识别未填、错填、冲突字段
- 结合架构图做补全与校验

适用场景：

- 客户已按旧流程准备过材料
- 需要兼容历史模板流转方式

---

## 四、AI 处理逻辑

### 4.1 无配置文档，仅有业务描述 + 架构图

系统逻辑：

1. 解析业务语言，提取场景关键词
2. 解析架构图，识别服务组件、部署层次、网络边界、依赖关系
3. 匹配现有组合服务 / 原子服务能力
4. 生成推荐申请草稿
5. 输出缺失项和需要用户确认的问题

目标：

- 降低用户从零填写字段的门槛
- 让用户先确认“方向对不对”，再补“参数够不够”

### 4.2 有配置文档 + 架构图

系统逻辑：

1. 以 Excel 配置文档作为显式输入
2. 以架构图作为环境与拓扑校验依据
3. 做字段完整性检查
4. 做交叉一致性检查
5. 输出合理性建议与风险提示

重点不是“重新生成一份”，而是：

- 看用户给的内容是否合理
- 是否与架构图一致
- 是否满足目标环境和 SLA 要求

### 4.3 仅有配置文档

系统逻辑：

1. 解析现有模板内容
2. 自动映射为系统内部结构化字段
3. 标出高风险缺失项
4. 引导用户补充图或补充业务上下文

### 4.4 AI 输出应固定为三层结果

每次分析建议都输出三层结果，而不是只给一句“推荐配置”：

1. 申请草稿
   - 可直接进入需求细化单 / 申请单

2. 缺失项清单
   - 明确哪些字段尚无法确定

3. 合理性建议
   - 判断是否存在高可用不足、容量不匹配、网络边界不清、安全/备份缺失等问题

## 五、页面交互建议

### 5.1 发起页结构

建议把发起页拆成 4 个区块：

1. 需求输入区
   - 业务目标
   - 场景类型
   - 优先级
   - 目标环境
   - 上线时间

2. 材料上传区
   - 上传架构图
   - 上传已有配置文档
   - 下载示例模板 / 查看填写说明

3. AI 解析区
   - 已识别组件
   - 推荐服务项
   - 待确认问题
   - 风险提示

4. 确认与导出区
   - 确认生成需求细化单
   - 导出 Excel / PDF 评审材料
   - 保存草稿

### 5.2 按钮与入口建议

建议保留两个上传按钮，但视觉权重要区分：

- `上传架构图`：主按钮，突出显示
- `上传配置文档`：次按钮，标注“可选，适用于已有模板材料”

同时保留：

- `下载示例模板`
- `查看填写示例`

但避免让“下载模板”成为页面第一动作。

### 5.3 交互策略

不建议把上传动作做成“全都选填但没有引导”的弱入口，而应给默认提示：

- 有架构图：优先上传架构图
- 没有架构图：直接描述业务需求
- 已有客户模板：可导入 Excel，系统会帮你识别和补全

## 六、与当前系统能力的衔接方式

### 6.1 与 `field dictionary` 的关系

`field dictionary` 继续负责标准字段治理，不直接暴露给普通用户。

作用：

- 统一系统内部字段口径
- 为 AI 结果回填提供标准字段目标
- 支撑不同服务场景的字段复用

### 6.2 与 `schema templates` 的关系

表单模板继续保留，但角色从“用户自己填写模板”转为“系统渲染不同场景的引导式表单布局”。

作用：

- 控制不同服务/组合服务的展示布局
- 承接 AI 已识别字段的回填
- 控制哪些字段需要继续追问

### 6.3 与 `configProfiles` 的关系

`configProfiles` 不只是服务目录里的推荐档案，还应成为 AI 推荐的候选输出基础。

作用：

- 当用户未给明确配置时，AI 可根据业务语言 + 架构图推荐匹配的 `configProfile`
- 当用户上传了配置文档时，可用来比对是否与现有档案匹配

### 6.4 与组合服务的关系

组合服务不应只管理交付流程，还应沉淀“典型场景下 AI 应该优先推荐的配置组合”。

建议新增能力：

- 组合服务对应的推荐输入场景
- 组合服务对应的架构图识别标签
- 组合服务对应的推荐配置模板 / 推荐档案集合

## 七、产品定位调整

结合现有讨论，建议明确以下定位变化：

### 7.1 发起页不是“模板填写页”

发起页应定位为：

- `需求细化工作台`
或
- `架构评审准备工作台`

而不是简单的：

- `配置模板填写页`

### 7.2 Excel 不是主单据，而是导出物

正式主单据应是系统中的：

- 需求细化单
- 资源申请单

Excel / PDF 更适合作为：

- 评审材料
- 外部沟通材料
- 审批附件
- 归档副本

### 7.3 AI 的价值点要前移

AI 不应只在“用户填完后生成推荐配置”这一刻出现，而应前移到：

- 输入理解
- 缺失信息追问
- 架构图识别
- 合理性校验

---

## 八、建议的任务拆分

### 8.1 建议新增 / 重定义任务

1. Portal 发起页输入策略升级
   - 支持业务描述、架构图上传、配置文档导入三种输入模式

2. 架构图解析驱动的申请初稿生成
   - 输出推荐服务项、候选配置、缺失项

3. Excel 模板能力降级为导入/导出组件
   - 导入历史材料
   - 导出评审材料

4. AI 合理性校验前移
   - 从原来的后置配置推荐，前移到需求细化阶段

5. 组合服务补充“推荐场景/推荐配置”元数据
   - 支撑架构图识别后的自动推荐

### 8.2 优先级建议

建议优先级从高到低如下：

1. 发起页输入策略升级
2. 架构图上传与 AI 解析
3. 需求细化单结构扩展
4. 配置文档导入
5. 导出 Excel / PDF 评审材料

---

## 九、最终建议

如果本期只能选一条主线，建议选择：

`业务描述 + 架构图 -> AI 生成需求细化稿 -> 用户确认 -> 导出材料`

同时保留辅助路径：

`已有 Excel -> 上传导入 -> AI 校验补全`

不建议继续把“不同需求对应不同 Excel 模板”作为产品主方案，因为这无法解决你当前最核心的问题：用户不知道怎么填、填完还要反复沟通。

真正应该被产品化的是“引导、识别、补全、校验”这条能力链，而不是 Excel 模板本身。
