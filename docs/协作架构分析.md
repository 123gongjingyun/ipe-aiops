# IPE/AIOps 协作架构分析

> 分析日期：2026-06-11  
> 分析对象：Portal 协作架构页（`/#/about/collab`）与当前平台实现的对齐关系

---

## 一、架构全景

```text
业务价值 / 数字化方针 / KPI
        ↓
业务部门 → 产品团队/POA → 需求入口 → IDP 统一门户
                                              ↓
        ┌─────────────────────────────────────┼─────────────────────────────────────┐
        │                                     │                                     │
   ① AI-SDLC                         ② IPE 平台服务交付主线                   ③ AIOps/SRE
   (应用与数据服务开发)                  (申请→架构评审→审批→前台受理→交付→归档) (运行保障)
        │                                     │                                     │
        └─────────────→ 共同进入运行态 ←──────┘                                     ↓
                                                                             监控→告警→诊断→恢复→优化→反馈
        ↑                                                                      ↑
        └──────────── DevSecOps / SOC 安全治理护栏（贯穿全生命周期）─────────────┘
```

## 二、IPE 平台服务交付主线的位置

### 2.1 层级定位

IPE 位于 **IDP 统一数智化交付门户** 的核心层，是连接 **AI-SDLC 开发** 与 **AIOps/SRE 运行保障** 的桥梁。

| 层级 | 职责 | IPE 的角色 |
|------|------|-----------|
| 战略层 | 业务价值 / KPI | 承接业务目标，通过服务交付实现价值度量 |
| 入口层 | IDP 统一门户 | 统一入口、统一流程、统一服务目录 |
| 主线层 | AI-SDLC / IPE / AIOps | **IPE 负责从申请到交付资产归档的交付闭环** |
| 护栏层 | DevSecOps / SOC | 安全治理贯穿 IPE 全生命周期 |
| 底座层 | 工具链 / 基础技术平台 | 为 IPE 提供编排、交付、配置能力 |

### 2.2 流程定位

架构图标准流程：

```
申请 → 架构评审 → ITSM / 资源审批 → 交付中心前台受理 → 实施方案确认 → 交付实施 → 验收归档
```

当前平台实际流程：

```
需求发起 → 架构评审 → ITSM / 资源审批 → 交付中心前台受理 → 实施方案确认 → 交付实施 → 验收归档
pending → reviewing → processing → plan_confirming → delivering → completed → confirmed → archived  （工单状态）
```

**当前结论**：评审环节已显式落地；后续重点不再是增加发起页字段，而是继续完善 ITSM / 资源审批后的交付中心前台受理、实施方案形成、资产台账沉淀和正式 CMDB 同步。

## 三、IPE 与其他模块的协作关系

### 3.1 上游协作

| 协作方 | 交付物 | IPE 的响应 |
|--------|--------|-----------|
| **业务部门 / 产品团队** | 服务交付申请 | 通过 Portal 接收申请，生成工单 |
| **IDP 统一门户** | 统一需求入口 | 承接门户分发的服务请求 |
| **AI-SDLC** | 已开发的应用/数据服务 | AI-SDLC 发布后的服务需要通过 IPE 进行平台级交付编排 |

### 3.2 下游协作

| 协作方 | IPE 的交付物 | 接收方的响应 |
|--------|-------------|-------------|
| **AIOps/SRE 运行保障主线** | 已注册的服务资产 | 进入运行态，开始监控、告警、诊断、恢复 |
| **统合运维中心（大一线）** | 交付完成的工单 | 接手事件、问题、变更、发布管理 |
| **交付资产池** | 已归档的交付资产 | 形成可查询、可复用的资产沉淀 |

### 3.3 横向协作

| 协作方 | 协作方式 |
|--------|----------|
| **DevSecOps / SOC** | 贯穿全生命周期的安全护栏：身份权限 → 代码安全 → 数据安全 → 漏洞扫描 → 合规审计 → 安全监测 → 事件响应 |
| **架构治理团队** | 方案评审阶段需经过架构治理团队的 4A 原则评审 |
| **平台工程团队** | 提供平台建设、工具链集成、服务目录维护 |

### 3.4 底座协作

| 底座组件 | IPE 的使用方式 |
|----------|---------------|
| **工具链与共通服务** | AI 工具集（编排推荐）、API 网关（服务注册）、开发/测试工具集 |
| **基础技术平台** | 云、容器、DB、中间件、网络、CMDB、监控、日志、机房、AI 算力 |
| **配置数据字典** | 原子服务的输入输出 Schema、组合服务的配置模板 |

## 五、关键缺口与优化方向

### 方向 1：IPE 主线已补齐评审，继续增强审批与交付前台受理

**现状**：当前流程已从旧的「申请/接单→编排/方案确认」调整为「需求发起→架构评审→ITSM / 资源审批→交付中心前台受理→实施方案确认→交付实施→验收归档」。

**已落地**：
- `reviewing`（评审中）状态已加入工单状态流。
- 架构评审与 ITSM / 资源审批已拆分表达。
- ITSM / 资源审批通过后，不再直接进入用户确认，而是进入交付中心前台正式受理。
- Center 可维护标准/非标路径、能力域和正式实施方案，再提交用户确认。

**当前状态流**：
```
pending → reviewing → processing → plan_confirming → delivering → completed → confirmed → archived
```

**后续增强**：
- 审批流程配置页面。
- 真实 ITSM 集成和审批结果自动回传。
- 交付资产台账到正式 CMDB 的同步状态。

### 方向 2：多角色治理基础已落地，继续增强菜单级权限

**现状**：当前平台已在设置页提供“角色与用户”入口，支持角色种子、成员维护、本地持久化和搜索；审批节点也已按角色表达。

**待增强**：菜单级权限控制、真实账号体系、审计记录和外部统一认证仍需继续建设。

**建议角色体系**：

| 角色 | 职责 | 可见菜单 |
|------|------|----------|
| **平台管理员** | 系统配置、用户管理、服务目录维护 | 全部 |
| **交付工程师** | 工单处理、方案编排、交付执行 | 工单管理、服务目录、交付资产、运维集成中心 |
| **申请人（业务用户）** | 提交申请、确认方案、验收服务 | 服务目录、我的申请、平台概览 |
| **评审人 / 架构师** | 方案评审、架构治理 | 工单管理（评审视角）、能力矩阵 |
| **运维人员** | 运行保障、故障处理 | 运维集成中心、交付资产 |
| **安全管理员** | 安全治理、合规审计 | 安全运维、工单管理 |

### 缺口 3：原子服务输入输出未独立为数据字典

**问题**：当前 `inputSchema` / `outputSchema` 硬编码在每个原子服务定义文件中（如 `compute.ts`、`database.ts`），没有独立的管理页面。

**影响**：
- 新增/修改字段需要改代码文件
- 无法复用相同字段定义
- 不支持非开发人员维护配置数据

**建议方案**：
- 新增「数据字典」独立页面（建议在 Center 服务目录下或作为一级菜单）。
- 数据字典管理：字段名、标签、类型、选项、默认值、是否必填。
- 原子服务编辑时从数据字典中选择字段，而不是硬编码。

### 缺口 4：安全运维的贯穿性表达不足

**问题**：架构图中 DevSecOps/SOC 是「贯穿全生命周期」的护栏，但当前安全运维仅作为运维集成中心的一个运行域。

**分析**：
- 当前「安全运维」定位为运行侧安全（漏洞、WAF、审计、威胁检测），这本身是合理的。
- 架构图中的「贯穿性」更多是指安全治理流程（身份权限、代码安全、数据安全等）需要在各阶段都有检查点，而不是必须在 UI 上跨层展示。

**建议**：
- **保持当前安全运维入口不变**，已有独立页面和菜单，满足运行侧安全需求。
- 后续如需强化「贯穿性」，可在以下位置增加安全相关的状态提示或快捷入口：
  - 工单详情页增加「安全合规检查」步骤
  - 交付资产页增加「安全配置审计」标签
  - 运维集成中心总览页增加「安全水位」KPI
