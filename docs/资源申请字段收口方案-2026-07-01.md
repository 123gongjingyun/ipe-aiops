# 资源申请字段收口方案

> 创建日期：2026-07-01  
> 适用范围：`project-mywayaiops-refactor-V1` Portal 发起页 / Center 评审节点 / Shared 数据模型  
> 目标：把 `【资源申请】基础资源需求_V1.xlsx` 的有效信息拆解为可产品化的字段模型，同时保持当前工单主流程状态判断不变

---

## 一、适用前提

本方案只处理“资源申请模板的字段承载方式”和“页面字段归类”，不改变当前工单主流程状态机：

`pending -> reviewing -> processing -> plan_confirming -> delivering -> completed -> confirmed -> archived`

因此：

- 本方案默认只扩展 `answers` / `extras` / 附件元数据 / 评审补充信息
- 不把普通表单字段直接作为流程状态推进条件
- 如后续确有字段需要参与流程门禁，必须单独评审并补测试

---

## 二、字段承载分层

建议把 Excel 里的字段拆成下面 5 层：

1. 发起页主表单字段
2. 系统推荐字段
3. 用户可调字段
4. 评审/审批补充字段
5. 交付/运营回填字段

字段设计优先级遵循：

1. 先保证业务意图可表达
2. 再保证资源套餐可推荐
3. 再保证评审和交付有补充入口
4. 最后才考虑导出格式与历史 Excel 对齐

---

## 四、系统推荐字段

这部分不建议要求用户从空白开始填写，应由系统根据应用类型、环境、业务级别和架构图分析结果自动生成初稿。

### 4.1 容器资源推荐

| 字段key | 中文名称 | 说明 |
|---|---|---|
| `resourcePartition` | 资源分区 | 如 `LAN` / `DMZ` / 专项资源池 |
| `instanceCount` | 实例数 | 标准推荐，可调 |
| `cpu` | CPU | 标准推荐，可调 |
| `memory` | 内存 | 标准推荐，可调 |
| `resourceRemark` | 资源备注 | 系统给默认说明，用户可补充 |

## 五、用户可调字段

为了避免“推荐即锁死”或“全部自由填写”两个极端，建议只开放少量高频参数给用户调整：

| 字段key | 中文名称 | 调整建议 |
|---|---|---|
| `instanceCount` | 实例数 | 可改，超边界触发非标 |
| `cpu` | CPU | 可改，超边界触发非标 |
| `memory` | 内存 | 可改，超边界触发非标 |
| `systemDiskSize` | 系统盘 | 可改 |
| `dataDiskSize` | 数据盘 | 可改 |
| `backupPolicy` | 备份策略 | 可改 |
| `retentionPeriod` | 保留周期 | 可改 |
| `resourcePurposeDetail` | 用途补充 | 可改 |
| `resourceRemark` | 备注 | 可改 |

建议保留一条产品规则：

- 在推荐套餐范围内调整，仍视为标准申请
- 超出预设边界后，标记为“非标申请”，进入评审节点重点确认

## 八、页面分组建议

如果后续做发起页原型，建议按下面结构组织，而不是按 Excel sheet 拆页面。

### 8.1 页面区块

1. 基础信息
2. 需求与目标
3. 网络与访问
4. SLA 与安全策略
5. 架构图与附件
6. 系统推荐资源
7. 用户调整与补充说明

### 8.2 页面原则

- 同一页面最多先呈现 25 个左右主字段，避免原 Excel 大表平移
- 推荐结果和用户输入要分区展示，避免用户误解“系统推荐就是已提交结果”
- 非标项需显式标记，不要隐藏在备注里
- 架构图上传应独立成明显区块，不与普通附件混放

---

## 九、与当前代码模型的关系

本方案与现有项目结构的关系如下：

- 已有主输入字段，优先继续落在 `order.answers`
- 补充性布尔/备注字段，可放在 `order.answers` 或 `order.extras`
- 附件类信息走附件元数据，不进入流程状态判断
- 评审类字段进入评审快照或节点补充结构
- 交付结果类字段继续落到 `serviceProgress.deliveryDetail` / 交付资产结构

当前阶段默认不做：

- 直接改 `workflow-stage.ts`
- 直接改 `order-stage-flags.ts`
- 直接改 `order-stage-nodes.ts`
- 直接把 Excel sheet 名称映射为流程阶段

---

## 十、实施顺序建议

建议按下面顺序推进，而不是一次性改完所有字段：

1. 先统一文档口径
2. 先收口发起页主字段
3. 再引入推荐套餐字段
4. 再补评审/运营回填字段
5. 最后才处理导出表头和历史 Excel 对齐

优先级最高的一批新增字段建议是：

- `integrationSystems`
- `userTypes`
- `clientTypes`
- `resourcePartition`
- `resourcePurpose`
- `databaseName`
- `charset`
- `logRetentionDays`
- `requiresBackup`
- `restoreRequirement`
- `rtoTarget`
- `rpoTarget`
- `drRequirement`
- `requiresPipelineAccess`
- `requiresEfkAccess`
- `requiresMonitoring`
- `requiresLogging`

这些字段优先补齐后，已经足以承接大部分 Excel 模板的有效信息。
