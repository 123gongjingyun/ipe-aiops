## 2.2 当前运行模式

当前运行模式仍是：

- `Portal` 和 `Center` 两个独立 SPA
- 通过 `shared` 共用类型和运行时逻辑
- 通过 `localStorage + BroadcastChannel` 共享数据
- 通过开发态 `/api/dev/*` 做模拟同步

这意味着当前是：

- **高保真原型**
- **非正式多用户在线系统**

这也是后续要引入 backend 和数据库的直接原因。

## 5. Portal 表单实现链路详细说明

## 5.1 当前申请流类型

Portal 当前主要有 3 类申请流：

1. 组合服务申请
2. 原子服务申请
3. 互联网应用申请

入口判断在：

- [apply.tsx](/Users/gjy/project-mywayaiops-refactor-V2/src/packages/aiops-service-portal/src/pages/apply.tsx:1)

## 5.2 表单字段来源

Portal 表单字段不是完全写死在页面里，而是运行时解析：

1. 根据服务规格取到 spec
2. 根据 spec 找到模板 / 模板版本
3. 根据模板版本解析 fields 和 layout
4. 用通用渲染器渲染

对应关键方法：

- `getResolvedSpecSchemaFields`
- `getResolvedSpecSchemaLayout`

## 5.3 表单布局来源

布局配置由 `FormLayoutConfig` 提供，当前支持：

- section
- row
- column
- span
- visibleWhen

渲染组件：

- [template-form-layout.tsx](/Users/gjy/project-mywayaiops-refactor-V2/src/packages/aiops-service-portal/src/components/template-form-layout.tsx:1)

这意味着当前已经具备未来“在线布局编辑器”的数据基础。

## 5.4 Portal 层承担的额外职责

Portal 不是只负责渲染字段，还承担：

- 表单值初始化
- 默认值带出
- 业务字段联动
- AI 推荐生成
- 附件处理
- 提交后生成工单 / 申请记录
- 快照生成

所以后续不能简单理解为“把 Portal 表单做成纯配置驱动，一点业务代码都不要”。

正确边界是：

- 字段、模板、布局、套餐配置数据：在线维护
- 复杂业务逻辑和流程控制：代码维护

这是后续开发必须一直保持清晰的产品边界。

当前 `资源申请单` 页面定位是：

- 发起前材料记录区
- 草稿保存和回溯区

## 9. 后续详细技术方案

## 9.1 后端方案

建议新建 backend 服务，技术栈：

- NestJS
- TypeORM
- JWT
- 开发 SQLite
- 生产 MySQL

原因：

1. 参考项目 `project-mygetpre` 已有可借鉴实现
2. 适合后续接登录权限和用户体系
3. 适合承载配置中心和工单持久化

## 9.2 数据持久化对象

后端建议优先持久化两大类数据：

### A. 业务单据类

- users
- orders
- order_snapshots
- order_components
- order_component_fields
- order_attachments
- order_approval_notes
- order_export_records
- order_operation_logs

### B. 配置治理类

- field_dictionary
- schema_templates
- schema_template_versions
- schema_template_layouts
- config_profile_groups
- config_profiles
- template_config_profile_bindings
- service_spec_template_bindings
- roles
- menu_permissions

## 9.3 为什么必须有快照

因为后续明确要求：

- 历史复制
- 重新发起
- HTML / PDF / Excel 导出
- 审批意见补录
- 再次导出

如果没有快照，就会出现：

- 当前模板变了，历史导出口径无法回溯
- 原单内容和再发起内容无法区分
- 审批补录后的版本无法追踪

所以必须存：

- 申请单当前值
- 申请单快照
- 使用时模板版本
- 使用时布局版本

## 11. 导出体系详细设计

## 11.1 统一思路

导出不能继续分散实现，必须先建立统一中间模型。

统一中间模型建议字段：

- requestId
- requestTitle
- applicantInfo
- userRequirements
- requestSections
- componentSections
- attachmentSummary
- reviewSummaryOverview
- reviewTalkingPoints
- approvalNote
- placeholderHints

## 11.2 HTML 评审版

目标：

- 在线预览
- 作为 PDF 母版
- 支持后续分享和打印

结构：

1. 用户需求
2. 申请信息
3. 评审概要概览
4. 评审交流话术建议
5. 审批意见

要求：

- 空字段不隐藏
- 用轻量提示文案占位
- 保持表单感和审阅感
- `评审概要概览` 在填写页和导出页都可以展示，但必须保持短摘要形态
- `评审交流话术建议` 只出现在导出材料中，不放在日常填写页主界面
- `评审交流话术建议` 在 HTML 评审页默认折叠，用户手动展开后查看
- HTML 评审页应预留 `复制话术` 交互位
- PDF 导出应支持“是否包含评审交流话术建议”的导出选项

## 11.2.1 用户需求填写规则

`用户需求` 区块后续按下面规则固定执行：

1. `用户需求` 整块定义为必填区
2. 粉色标记字段由项目组填写
3. 除 `背景` 和 `网络需求-用户访问方式` 之外，其余“不涉及”的字段允许填写 `无`
4. `背景` 与 `用户访问方式` 必须填写具体内容，不可填写 `无`
5. 页面顶部统一放置填写规则提示
6. `重要总原则` 作为 `用户需求` 区块顶部的强提示规则显示，不挂在单个字段下

建议规则提示文案方向：
