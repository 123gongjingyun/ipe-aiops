# 能力矩阵自动化等级判定说明

## 1. 文档目的

本文用于说明 `Center / 能力矩阵` 页面中 `手工 / 半自动 / 全自动` 三种自动化等级的当前判定逻辑、现有限制，以及后续建议的改造方向。

适用范围：

- 本地开发环境
- 当前 `project-mywayaiops-refactor-V1` 原型版本
- Center 页面中的能力矩阵总览与详情页

---

## 2. 当前代码位置

当前自动化等级判定函数位于：

- `src/packages/shared/src/data/specs/capability-meta.ts`

核心实现如下：

```ts
export function deriveAutomationLevel(spec: AtomicServiceSpec): AutomationLevel {
  if (spec.automationLevel) return spec.automationLevel;
  if (spec.deliveryMode === 'ai') return 'full';

  const text = `${spec.name} ${spec.category} ${spec.description}`;
  if (text.includes('配置') || text.includes('接入') || text.includes('策略') || text.includes('开通')) {
    return 'semi';
  }
  return 'manual';
}
```

---

## 3. 当前判定逻辑

当前逻辑采用“显式字段优先，关键词兜底”的方式，优先级如下。

### 3.1 第一优先级：显式指定 `automationLevel`

如果某个原子服务 `spec` 自身已经带有 `automationLevel` 字段，则直接使用该值，不再继续推导。

示例语义：

- `manual`：手工
- `semi`：半自动
- `full`：全自动

这意味着：

- 如果服务目录元数据里已经明确维护了自动化等级，能力矩阵会完全信任该字段
- 当前页面不会再去检查这个服务的 `deliveryMode`、`handler`、执行方式等其他信息

### 3.2 第二优先级：`deliveryMode === 'ai'`

如果没有显式 `automationLevel`，但 `spec.deliveryMode === 'ai'`，则直接判定为：

- `full`

当前语义可以理解为：

- 该服务由 AI 或自动编排主导交付
- 在能力矩阵中按“全自动”处理

### 3.3 第三优先级：关键词兜底判断

如果既没有 `automationLevel`，也不是 `deliveryMode === 'ai'`，则会把以下文本拼起来：

- `spec.name`
- `spec.category`
- `spec.description`

然后检查是否包含这些关键词：

- `配置`
- `接入`
- `策略`
- `开通`

只要命中任意一个关键词，就判定为：

- `semi`

否则，判定为：

- `manual`

---

## 4. 当前三种等级在原型中的实际含义

需要强调的是：当前这三档等级，更接近“能力运营标签”，而不是“真实交付执行事实”。

### 4.1 手工 `manual`

当前含义：

- 没有显式配置自动化等级
- 不是 AI 自动交付
- 名称/分类/描述中也没有命中“配置、接入、策略、开通”等关键词

在当前原型里，`manual` 更像默认值。

### 4.2 半自动 `semi`

当前含义：

- 没有显式配置自动化等级
- 不是 AI 自动交付
- 但名字或描述上看起来像“平台可以辅助做一部分”的服务

典型被归为半自动的文字特征：

- 配置类
- 接入类
- 策略类
- 开通类

注意：

- 这并不代表系统真的已经具备半自动执行能力
- 只是原型当前把这类服务视作“自动化潜力较高、可平台辅助”的一类

### 4.3 全自动 `full`

当前含义：

- 服务元数据明确标记为自动化等级，或
- `deliveryMode === 'ai'`

在原型中，它表示该服务被归为“自动执行能力最强的一档”。

但它仍然不是对真实执行链路的严格证明。

---

## 5. 当前能力矩阵如何使用这个等级

### 5.1 在总览页中的用法

总览矩阵中，每个领域、每种服务形态的格子会统计：

- 总服务数 `total`
- 已上线数 `online`
- 自动化数 `automated`

其中 `automated` 的当前逻辑是：

- 只要 `deriveAutomationLevel(spec) !== 'manual'`
- 就记入自动化覆盖

也就是说：

- `semi` 计入自动化覆盖
- `full` 计入自动化覆盖
- `manual` 不计入自动化覆盖

因此总览页展示的“自动化覆盖”本质上是：

> 非手工服务占比

而不是：

> 真实自动执行比例

### 5.2 在详情页中的用法

详情页会分别统计：

- `manual`
- `semi`
- `full`

三档分布，并用于展示该领域的自动化等级结构。

---

## 6. 当前逻辑存在的问题

当前逻辑能满足原型展示，但从产品定义和后续演进角度看，问题比较明显。

### 6.1 关键词判断过于粗糙

例如：

- 名字里有“开通”，就会被判成半自动
- 但现实中很多“开通”其实仍然是纯人工工单流

同样地：

- 名字里没有“配置/接入/策略/开通”
- 但实际上已经可以由脚本或平台自动执行
- 当前仍可能被判成手工

这会造成展示失真。

### 6.2 没有真正使用交付能力元数据

当前判定没有使用这些更接近真实能力的字段：

- `delivery.handler`
- `delivery.dependencies`
- `delivery.autoDays`
- `delivery.manualDays`
- 是否有真实自动执行器
- 是否需要人工审批后继续
- 是否需要跨系统人工处理

因此，它不是按真实能力建模，而是按静态描述猜测。

### 6.3 很多服务默认会落到 `manual`

当前大量 spec 的 `deliveryMode` 仍然是 `manual`。

这意味着：

- 即便某些服务底层已经有自动化处理器名字，例如 `vm-provision`、`db-provision`
- 只要没有显式维护 `automationLevel`
- 最终仍可能被归到 `manual`

### 6.4 枚举值可能存在历史口径不一致

仓库里还能看到一些历史值，例如：

- `semi-auto`

而当前能力矩阵使用的标准返回值是：

- `manual`
- `semi`
- `full`

这说明当前自动化等级元数据口径没有完全统一，后续需要收敛。

---

## 7. 为什么当前先这样做

从原型角度看，这套逻辑是有它存在理由的：

1. 页面需要先有一个自动化分层结果，便于能力矩阵展示。
2. 当前服务目录元数据还没有完整维护统一的自动化等级字段。
3. 真实交付执行链路和外部系统能力还没有全部接通，没法依据运行事实精准统计。
4. 先用“显式字段 + 关键词兜底”可以快速形成一个可展示的运营视图。

因此，当前实现适合：

- 原型展示
- 方案沟通
- 能力总览

但不适合直接作为：

- 正式 KPI
- 真正的自动化成熟度指标
- 对外承诺口径

---

## 8. 建议的正确产品语义

建议后续把三档语义明确成下面这样：

### 8.1 手工

定义：

- 主要交付动作依赖人工执行
- 平台只能做记录、提示、汇总

### 8.2 半自动

定义：

- 平台可以自动生成参数、调用部分接口、执行部分步骤
- 但关键节点仍依赖人工确认、人工审批或人工跨系统处理

### 8.3 全自动

定义：

- 平台可以自动完成主要交付动作
- 人工主要在审批、异常处理、审计复核时介入

---

## 9. 建议的后续改造方向

### 9.1 改为显式维护 `automationLevel`

建议不要长期依赖关键词推导，而是在服务目录元数据中显式维护：

- `manual`
- `semi`
- `full`

能力矩阵只消费这个显式字段。

### 9.2 统一历史枚举值

需要把历史数据里可能存在的：

- `semi-auto`
- `full-auto`
- 其他旧写法

统一映射成：

- `manual`
- `semi`
- `full`

避免前后端口径不一致。

### 9.3 引入更真实的判定依据

如果后续要提高可信度，可以把以下信息纳入判定或展示：

- 是否存在真实自动执行器
- 是否调用外部平台 API
- 是否需要人工审批
- 是否依赖人工跨系统配置
- 是否具备全流程回滚与审计

这样能力矩阵的自动化等级才会更接近真实交付成熟度。

### 9.4 区分“定义等级”和“运行事实”

建议未来把两个概念拆开：

- 自动化等级：服务定义层面的能力标签
- 自动化达成率：运行时实际自动执行比例

前者用于能力矩阵，
后者用于运营 KPI 和持续优化。

---

## 10. 当前结论

当前本地环境中的能力矩阵自动化等级逻辑，可以总结为：

> 优先使用服务元数据中的显式 `automationLevel`；如果没有，则 `deliveryMode === 'ai'` 判为全自动；否则通过“配置 / 接入 / 策略 / 开通”等关键词粗略判为半自动，其余默认判为手工。

因此当前页面展示的：

- `手工`
- `半自动`
- `全自动`

更准确的理解应当是：

> 原型阶段的能力运营标签

而不是：

> 真实生产环境下的自动化执行事实

---

## 11. 建议讨论结论

建议后续在产品和实现上统一采用下面原则：

1. 能力矩阵不再依赖关键词推导作为长期方案。
2. 自动化等级由服务目录显式维护。
3. 历史枚举值统一收敛。
4. 后续如需做成熟度运营，再增加基于真实执行链路的统计指标。

这样后续无论是对内汇报、对外展示，还是做能力建设路线图，口径都会更稳定。
