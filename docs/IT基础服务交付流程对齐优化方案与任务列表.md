# IT基础服务交付流程对齐优化方案与任务列表

## 1. 背景

根据 `IT基础服务交付流程改善.pptx` 以及当前系统流程讨论，当前平台与 PPT 方向整体一致：

- 用户通过统一入口发起基础服务需求
- 平台提供服务目录、套餐、原子服务和 AI 推荐
- 交付侧通过流程、编排和资产沉淀支撑交付闭环

但在流程责任边界上需要进一步澄清：

- 用户发起阶段已经具备结构化表单、AI 推荐和咨询入口，不建议继续增加大量前台沟通字段
- 架构评审由客户与供应商架构师共同完成，不应和交付中心前台受理混在一起
- ITSM / 资源审批通过之后，才应进入 IPE 交付中心前台正式受理
- 交付中心前台正式受理后，应形成实施方案，再提交用户确认，之后进入交付实施

因此，本次优化重点不放在发起页，而放在：

> ITSM / 资源审批通过后的交付中心前台正式受理与实施方案形成。

---

## 2. 推荐目标流程

建议将对外流程口径调整为：

```text
1. 需求发起
2. 架构评审
3. ITSM / 资源审批
4. 交付中心前台正式受理
5. 实施方案确认
6. 交付实施
7. 用户验收
8. 资产沉淀与归档
```

对应当前系统状态：

| 系统状态 | 建议业务含义 |
|---|---|
| `pending` | 用户发起需求，必要时可通过咨询入口沟通 |
| `reviewing` | 客户与供应商架构师进行架构评审 |
| `processing` | ITSM / 资源审批处理中 |
| `processing -> plan_confirming` | 审批通过后，交付中心前台正式受理并形成实施方案 |
| `plan_confirming` | 用户确认实施方案 |
| `delivering` | 交付实施 |
| `completed` | 交付完成，待用户验收 |
| `confirmed` | 用户已验收 |
| `archived` | 交付资产沉淀并归档 |

---

## 3. 产品原则

### 3.1 发起页保持轻量

发起页当前已经具备：

- 结构化需求表单
- AI 推荐方案
- 咨询入口

因此不建议新增：

- 需求咨询记录
- 前台建议套餐
- 前台建议补充材料
- 前台协助选择服务目录

原因：

- 会增加用户填写负担
- 容易让用户误以为必须先人工咨询才能发起
- 会把咨询辅助动作和正式流程记录混在一起

建议口径：

> 发起阶段通过结构化表单、AI 推荐和咨询入口完成轻量前台引导，不额外增加申请表单负担。

### 3.2 交付受理后置

交付中心前台正式受理不应放在架构评审前，也不应放在用户发起阶段。

正式交付受理应发生在：

```text
架构评审通过
→ ITSM / 资源审批通过
→ 交付中心前台正式受理
```

这个阶段要回答的是：

- 谁来交付
- 怎么交付
- 标准交付还是非标交付
- 涉及哪些能力域
- 需要哪些协同节点
- 实施步骤和计划是什么
- 风险和前置条件是什么

### 3.3 方案确认前必须有实施方案

用户确认的不应该只是抽象的 AI 推荐结果，而应该是审批通过后由交付中心前台正式受理并整理后的实施方案。

实施方案应包含：

- 服务清单
- 涉及能力域
- 标准 / 非标路径
- 实施步骤
- 预计时间
- 前置条件
- 风险说明
- 交付产出

---

## 4. 目标产品形态

### 4.1 Portal 用户侧

用户侧不增加复杂表单字段。

需要增强的是工单详情展示：

- 架构评审状态
- ITSM / 资源审批状态
- 交付中心前台是否已受理
- 实施方案是否待确认
- 实施方案核心内容

用户在 `plan_confirming` 阶段看到的是：

> 审批已通过，交付中心前台已受理，并形成实施方案，请确认后进入交付。

### 4.2 Center 运营侧

Center 是本次优化重点。

在工单详情中，应把 `processing` 阶段拆清楚：

1. ITSM / 资源审批
2. 审批通过后的交付中心前台正式受理
3. 实施方案形成
4. 提交用户确认

运营侧需要具备以下动作：

- 更新审批状态
- 标记交付中心前台已受理
- 选择标准 / 非标交付路径
- 选择涉及能力域
- 生成或编辑实施方案
- 提交方案给用户确认

## 5. 建议数据模型

建议在 `Order` 上增加一个交付受理与实施方案快照。

### 5.1 建议字段

```ts
deliveryAcceptance?: {
  status: 'not_started' | 'accepted' | 'plan_ready';
  acceptedBy?: string;
  acceptedAt?: string;
  deliveryPath?: 'standard' | 'non_standard';
  domains?: string[];
  implementationPlan?: {
    summary: string;
    steps: Array<{
      name: string;
      owner: string;
      mode: 'manual' | 'ai' | 'hybrid';
      output: string;
    }>;
    prerequisites: string[];
    risks: string[];
    estimatedSchedule: string;
    deliverables: string[];
  };
};
```

### 5.2 字段语义

- `status`
  - 当前交付受理状态
- `acceptedBy`
  - 交付中心前台受理人或受理组织
- `deliveryPath`
  - 标准交付或非标交付
- `domains`
  - 涉及能力域，例如 IT基础、网络、机房、安全、账号、监控
- `implementationPlan`
  - 审批通过后形成的正式实施方案

## 6. 任务列表

### P0：统一流程口径

目标：

- 先修正文案和流程理解，避免继续把“前台咨询”和“正式交付受理”混在一起。

任务：

- [x] 更新 Portal 帮助中心流程说明
- [x] 更新 Center 帮助中心流程说明
- [x] 更新流程训练页/标准交付流程文案
- [x] 更新 README 中流程口径
- [x] 更新对外沟通文档中的流程描述

验收标准：

- 文档与页面中统一表达为：
  - 发起阶段支持轻量咨询
  - 正式交付受理发生在 ITSM / 资源审批通过后
  - 实施方案由交付中心前台形成并提交用户确认

### P1：补交付中心前台正式受理数据模型

目标：

- 为审批通过后的交付中心前台受理和实施方案提供结构化承载。

任务：

- [x] 在 `Order` 类型中增加 `deliveryAcceptance`
- [x] 在订单 Store 中补默认初始化逻辑
- [x] 在历史订单 normalize 逻辑中补兼容
- [x] 增加更新交付受理状态的方法
- [x] 增加更新实施方案的方法

验收标准：

- 旧订单不报错
- 新订单可记录交付受理人、标准/非标路径、能力域和实施方案
- Store 单元测试覆盖基础读写和历史兼容

### P2：Center 工单详情增加交付中心前台受理区块

目标：

- 在运营侧明确表达审批通过后的正式交付受理。

任务：

- [x] 在 Center 工单详情 `processing` 阶段增加“ITSM / 资源审批”展示
- [x] 审批通过后展示“交付中心前台正式受理”区块
- [x] 支持选择标准 / 非标交付路径
- [x] 支持选择涉及能力域
- [x] 支持生成或编辑实施方案
- [x] 支持提交实施方案进入 `plan_confirming`

验收标准：

- `processing` 阶段不再泛泛显示“处理中”
- 用户可以看到审批与交付中心前台受理是两个连续动作
- 交付方案提交后，工单进入 `plan_confirming`

### P3：Portal 工单详情展示实施方案

目标：

- 用户侧在方案确认阶段看到的是正式实施方案，而不是纯系统推荐摘要。

任务：

- [x] 在 Portal 工单详情展示交付中心前台受理状态
- [x] 在 `plan_confirming` 阶段展示实施方案摘要
- [x] 展示能力域、实施步骤、预计时间、风险和交付产出
- [x] 保留确认方案和反馈意见动作

验收标准：

- 用户能清楚看到方案来源于交付中心前台正式受理
- 用户确认后进入 `delivering`
- 用户反馈后可回到 Center 修改方案

### P4：标准 / 非标路径可视化

目标：

- 对齐 PPT 中标准类和非标类需求治理思路。

任务：

- [x] 在 Center 交付受理区块中增加标准 / 非标判定
- [ ] 标准类展示套餐和原子服务编排
- [ ] 非标类展示方案设计、服务定制、能力域协同
- [ ] 在工单列表中增加路径标签
- [ ] 在能力矩阵中预留非标能力建设入口

验收标准：

- 标准类和非标类工单在列表和详情中可区分
- 非标类不影响当前标准交付主流程
- 后续可以继续扩展非标方案设计流程

### P5：资产沉淀与 CMDB 口径统一

目标：

- 统一对外表达：当前 CMDB 录入先对应交付资产录入和资产台账沉淀。

任务：

- [ ] 在交付资产页补充“资产台账沉淀”说明
- [ ] 在工单归档节点中明确“资产已沉淀”
- [ ] 在文档中统一 `CMDB录入 = 交付资产录入 / 后续可同步正式CMDB`
- [ ] 预留 `cmdbSyncStatus` 字段设计文档

验收标准：

- 对外沟通不再表述为“缺少 CMDB”
- 系统表达为“已具备交付资产台账，后续可扩展 CMDB 同步”

## 7. 推荐实施顺序

建议按以下顺序推进：

```text
P0 文案口径统一
→ P1 数据模型
→ P2 Center 交付中心前台受理
→ P3 Portal 方案确认展示
→ P4 标准/非标可视化
→ P5 资产/CMDB口径增强
```

原因：

- P0 成本最低，先统一团队理解
- P1 是后续页面改造基础
- P2 是本轮真正核心
- P3 是用户侧闭环
- P4 和 P5 可以作为增强迭代

---

## 8. 第一阶段最小可交付范围

如果要尽快做出一版可演示效果，建议第一阶段只做：

1. Center 工单详情增加“交付中心前台正式受理”区块
2. 支持标准 / 非标路径选择
3. 支持填写实施方案摘要、步骤、风险和交付产出
4. 提交后进入用户方案确认
5. Portal 方案确认页展示该实施方案

不建议第一阶段改发起页。

---

## 9. 对外沟通口径

建议对外这样描述本轮优化：

> 平台发起阶段保持轻量，通过结构化表单、AI 推荐和咨询入口完成前置引导；正式交付受理发生在架构评审和 ITSM / 资源审批通过之后，由 IPE 交付中心前台基于审批结果、服务目录、套餐和能力域情况形成实施方案，再提交用户确认并进入交付实施。这样既保证前端申请体验不被复杂流程干扰，也能把交付中心前台的正式受理、能力域协同和实施方案沉淀到平台流程中。

---

## 10. 当前结论

当前发起页基本不需要重构。真正需要优化的是审批通过后的交付中心前台正式受理环节。

本轮优化的核心应聚焦于：

```text
ITSM / 资源审批通过
→ 交付中心前台正式受理
→ 形成实施方案
→ 用户确认方案
→ 进入交付实施
```

这条链路补齐后，当前系统会更贴近 PPT 中“基础服务交付中心”的组织机制和交付运营方向。
