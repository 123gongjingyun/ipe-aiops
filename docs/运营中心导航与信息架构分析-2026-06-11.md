# 运营中心导航与信息架构分析

- 分析时间：2026-06-11
- 最近更新：2026-06-12
- 适用范围：`src/packages/aiops-service-center`
- 结论状态：已按当前方案落地

## 背景

本轮调整围绕两个目标展开：

1. 保持运营中心像公有云控制台一样清晰，但不要让用户感到“功能太多、成本太高”。
2. 给核心页面留出足够内容空间，避免侧栏和二级菜单持续膨胀。

本次已同步纳入后续规划中的 `套餐与定价`、`账号与安全` 等治理模块，确保导航不会因为后续能力扩张再次失控。

## 当前结论

### 1. 全局导航保持精简，复杂治理能力收口到页面内部

结论：

- `设置` 保持单一一级入口。
- `字段字典 / 配置模板 / 套餐与定价 / 角色与用户 / 审批流程 / 账号与安全` 统一收口到 `设置` 页面内部导航。
- 不再继续为治理类能力新增左侧全局二级菜单。

原因：

- 后续还会追加套餐设计、价格评估、流程联动等能力，如果继续平铺到全局导航，很快会超载。
- 治理类页面访问频次低于工单、目录、资产，适合收口为工作台。
- 用户右上角保留 `账号与安全 / 修改密码` 快捷入口，日常操作路径不会变长。

### 2. 能力矩阵保留一级入口，但详情下沉到二级页面

结论：

- `能力矩阵` 继续作为一级页面，承担平台能力全景视图。
- 领域级详细内容不在一级页平铺，而是通过 `/matrix/:domainKey` 二级页面承接。
- 一级页只保留总览、指标和下钻入口。

原因：

- 一级页同时堆叠矩阵、关联摘要、服务明细会明显挤压空间。
- 领域详情与工单、服务目录、资产的关联更适合在单独页面集中展示。
- 面包屑可承担回跳职责，不需要再在左侧增加一层导航。

### 3. 运维集成中心改为“单一入口 + 总览下钻”，不在左侧展开 5 个二级菜单

结论：

- 左侧导航仅保留一个 `运维集成中心` 一级入口。
- `监控中心 / 运维管理 / 安全运维 / 故障处理 / 机房设施` 不再作为左侧常驻二级菜单。
- 用户先进入 `运维集成中心` 总览页，再通过 4 层 5 域卡片进入具体页面。
- 具体页面统一通过顶栏面包屑回到 `运维集成中心`。

原因：

- 这 5 个域会让侧栏在视觉上显得“平台功能很多”，对初次进入用户有压迫感。
- 当前每个域页面内容深度仍有限，更适合从总览页承接，而不是长期占据主导航。
- 这种结构和能力矩阵的“一级总览 + 二级详情”模式一致，用户心智更统一。

### 4. 左侧导航采取公有云控制台式收敛布局

结论：

- 左侧导航宽度缩窄，默认优先给内容区留空间。
- 桌面端支持收起为图标态。
- 收起按钮放在左下区域，而不是顶部长条区域。
- 用户入口放在右上角，承接个人安全和快捷操作。

原因：

- 顶部区域应保留给品牌与主信息，不适合再叠加控制动作。
- 左下角放收起按钮更符合“辅助操作”的优先级。
- 右上角用户区符合公有云控制台常见习惯。

## 当前推荐结构

```text
运营中心
├── 仪表板
├── 工单管理
├── 服务目录
│   ├── 全部服务
│   ├── SLA管理
│   └── 服务流程
├── 能力矩阵
├── 交付资产
├── AI知识库
├── 运维集成中心
└── 设置
```

补充说明：

- `帮助中心` 作为辅助入口保留在导航底部，不参与主信息架构竞争。
- `运维集成中心` 放在 `AI知识库` 下方，优先级低于交付主链路页面。
- 后续新增 `套餐与定价`、`密码重置` 等治理功能，不再进入全局左侧导航。

## 页面分层

建议按“定义层 → 能力层 → 结果层 → 运行层 → 治理层”理解页面职责。

### 定义层

页面：

- `服务目录`
- `SLA管理`
- `服务流程`

职责：

- 回答平台提供什么服务。
- 回答按什么标准和流程交付。

### 能力层

页面：

- `能力矩阵`

职责：

- 回答平台能力覆盖到哪里。
- 回答每个技术域有哪些服务与能力沉淀。

### 结果层

页面：

- `交付资产`

职责：

- 回答已经交付并沉淀了什么。
- 提供按归档工单和按属性归类两个视角回查。

### 运行层

页面：

- `运维集成中心`
- `监控中心`
- `运维管理`
- `安全运维`
- `故障处理`
- `机房设施`

职责：

- 回答交付完成后由哪些运行域承接。
- 回答不同运行域的入口与职责边界。

### 治理层

页面：

- `设置`

职责：

- 回答平台怎么治理字段、模板、权限、流程、价格和账号安全。
- 通过页内导航承接大量低频治理功能，避免侵占全局主导航。

## 套餐与定价如何纳入

建议定位：

- 不独立进入全局主导航。
- 先收口到 `设置 -> 套餐与定价`。
- 后续在 Portal 申请、审批流程、工单详情中以“估算结果”形式透出，而不是先做一个高调主入口。

原因：

- 套餐与定价本质上是治理与评估能力，不是每天高频访问的主业务入口。
- 先放在设置内有利于逐步补齐规则、参数和模型。
- 等流程联动成熟后，再考虑是否升级为更独立的成本治理模块。

## 当前已落地的交互规则

1. 顶栏使用面包屑承担上下文返回，不再增加第二套全局顶部菜单。
2. `能力矩阵` 与 `运维集成中心` 都采用“一级总览页 + 二级详情页”的结构。
3. 用户相关操作放在右上角菜单，同时 `账号与安全` 仍保留在 `设置` 内部。
4. 左侧导航优先展示主链路页面，不让治理和运行域细项抢占首屏注意力。

## 一句话结论

运营中心采用“全局导航做减法，页面内部做组织”的策略：主导航只保留少量稳定一级入口，`能力矩阵` 与 `运维集成中心` 统一用总览页承接下钻，`设置` 作为治理工作台收口套餐、账号安全和流程能力，从而兼顾可扩展性与页面留白。
