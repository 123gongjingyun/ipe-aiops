# 资源申请字段数据模型草案

> 创建日期：2026-07-01  
> 适用范围：`project-mywayaiops-refactor-V1` Shared / Portal / Center  
> 目标：明确资源申请相关字段应落入哪一类数据结构，避免页面、导出、评审、交付各自散写一套口径

---

## 一、原则

本草案只定义字段归属，不改变当前主流程状态机：

`pending -> reviewing -> processing -> plan_confirming -> delivering -> completed -> confirmed -> archived`

字段归属原则：

1. 用户在发起页直接填写的内容，优先放 `order.answers`
2. 非核心、辅助、布尔开关或临时补充内容，可放 `order.extras`
3. 附件和模板引用，不放入普通文本字段，单独走附件元数据
4. 评审结论、专项意见、网络/安全补充，不混入 `answers`
5. 交付结果和资产结果，不回写到 `answers`

---

## 二、推荐的数据分层

建议把资源申请相关信息分为 6 层：

1. `order.answers`
2. `order.extras`
3. `order.attachments` / 材料元数据
4. `order.review*` / 评审补充结构
5. `order.itsm` / 外部单据关联
6. `order.serviceProgress[].deliveryDetail` / 交付结果与资产结果

如果后续确实需要新增结构，也建议围绕这 6 层扩展，而不是把所有字段继续平铺在 `Order` 顶层。

## 四、`order.extras` 建议承载字段

`extras` 更适合放：

- 非主路径的补充说明
- 仅用于前台展示或轻量提示的值
- 不要求严格结构化，但又不想丢失的信息

### 建议放入 `extras` 的字段

| 字段key | 中文名称 | 原因 |
|---|---|---|
| `resourceRemark` | 资源备注 | 轻量补充信息 |
| `architectureTemplateUrl` | 架构图模板下载地址 | 配置型/展示型，不属于申请人输入 |
| `architectureGuideUrl` | 架构图说明地址 | 配置型/展示型 |
| `diagramIncludesExternalSystems` | 图中是否含外联系统 | 附件辅助标签，可先放 extras |
| `diagramIncludesPublicAccess` | 图中是否含公网链路 | 附件辅助标签，可先放 extras |
| `diagramIncludesDataLayer` | 图中是否含数据库/中间件 | 附件辅助标签，可先放 extras |
| `diagramIncludesDr` | 图中是否含容灾/双活 | 附件辅助标签，可先放 extras |

### 处理建议

如果后续附件体系单独成熟，这些 `diagramIncludes*` 字段可以再迁移到附件元数据里；在原型期先挂 `extras` 更稳。

## 五、附件元数据建议承载字段

附件不建议继续伪装成普通文本字段，建议单独管理。

### 5.1 建议的附件类型

| 附件类型 | 说明 |
|---|---|
| `architecture_diagram` | 应用/基础架构图 |
| `config_template_import` | 历史 Excel 配置模板导入件 |
| `supporting_material` | 其他补充材料 |

### 5.2 附件元数据建议字段

| 字段key | 中文名称 | 说明 |
|---|---|---|
| `attachmentType` | 附件类型 | 如 `architecture_diagram` |
| `fileName` | 文件名 | 展示 |
| `fileSize` | 文件大小 | 展示/校验 |
| `uploadedAt` | 上传时间 | 展示 |
| `uploadedBy` | 上传人 | 审计 |
| `templateVersion` | 模板版本 | 架构图模板版本 |
| `parseStatus` | 解析状态 | 预留 AI / OCR / 图解析 |
| `source` | 材料来源 | 用户上传 / 模板导入 / 系统生成 |

### 5.3 `architectureDiagram` 的处理建议

页面层可以继续叫 `architectureDiagram`，但落库时建议作为：

- 一条 `attachmentType=architecture_diagram` 的附件记录

不要长期把它作为 `answers.architectureDiagram = "xxx.png"` 这种弱结构来保存。

## 六、评审补充结构建议

评审类字段不建议继续混在 `answers` 中，否则申请人输入和评审结论会互相污染。

### 6.1 建议新增统一评审补充结构

可考虑抽象为：

```ts
reviewExtensions?: {
  architecture?: { ... }
  network?: { ... }

  backup?: { ... }
```

## 八、交付结果字段建议归属

交付结果不应回写进申请字段，而应继续放入交付结果结构。

### 8.1 建议继续放在 `serviceProgress[].deliveryDetail`

| 字段key | 中文名称 | 建议归属 |
|---|---|---|
| `deliveryPlatform` | 云平台/虚拟化平台 | `deliveryDetail` |
| `deliveryHostname` | 主机名 | `deliveryDetail.network.hostname` |
| `deliveryInstanceName` | 实例名 | `deliveryDetail.asset.instance` |
| `deliveryNamespace` | 命名空间 | `deliveryDetail.namespace.name` |
| `deliveryResourceQuota` | 资源配额 | `deliveryDetail.namespace.resourceQuota` |
| `tenantProvisionStatus` | 租户开通状态 | `deliveryDetail` 或交付附加结果 |

### 8.2 对资产台账的影响

这些字段仍应以交付结果为源，再映射到交付资产字段，而不是反过来从申请字段生成最终台账。

## 十一、和当前主流程保护的关系

本草案默认遵守下面边界：

1. 只扩展字段模型，不改主流程状态机
2. `answers` 字段变化不应直接改变 Portal / Center 当前阶段判断
3. 任何字段要进入“流程门禁条件”，必须单独列项评审
4. 页面、导出、推荐、评审四个层次共享同一套字段 key，避免别名继续扩散

## 十二、建议下一步

从研发可执行性看，下一步最合适的是继续补一份：

`推荐套餐规则草案`

重点回答：

- 哪些字段决定推荐容器套餐
- 哪些字段决定推荐虚机/数据库套餐
- 哪些情况下判定为非标
- 哪些情况下只给建议、不自动生成推荐

这样页面、字段和推荐逻辑三层就能闭合。
