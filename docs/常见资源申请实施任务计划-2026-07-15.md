# 常见资源申请实施任务计划（2026-07-15）

## 一、当前目标

先做一个可以直接拿去和用户对方向的版本，先解决“从哪里开始、怎么填、怎么导出”这三个问题。

当前阶段重点：

1. 先把页面结构调顺
2. 先把字段边界讲清楚
3. 先把高频资源申请入口独立出来
4. 先保留旧服务目录，不做硬替换
5. 先把页面信息层级、入口权重和填写说明逻辑调顺，方便后续继续和用户对齐

---

产出：

1. 首页结构调整
2. 左侧菜单调整
3. 旧服务目录降级
4. 页面头部区统一

当前已完成：

2. 首页已收口为“主入口在上、其他入口说明在下”
3. `填写说明` 已从一级菜单移除
4. `填写说明` 已改为常见资源申请页和填写工作区中的弹框辅助入口
5. 页面正文首屏已改为动作型表达，不再和页面头部标题重复

状态：已完成首版

当前结论：

1. 左侧导航收口为 5 个主入口 + 帮助中心
2. `填写说明` 只作为辅助弹框，不再占一级菜单
3. 旧服务目录保留，但只作为弱入口
4. 页面头部区与正文首屏的角色已经分开

### 任务 2：字段映射总表

目标：

1. 提炼常见资源申请的用户可见字段
2. 明确哪些来自参考平台
3. 明确哪些保留 Portal 当前材料整理能力

产出：

1. 字段差异分析
2. 字段映射总表

状态：已完成首版

当前结论：

1. 资源申请不直接照搬参考平台整页逻辑
2. 需要把参考平台字段映射成用户能理解的申请字段
3. Portal 现有材料整理能力保留，后续按字段继续补齐

2. 容器申请
3. OBS 申请
4. SFS 申请
5. 用户权限申请
6. 网络策略申请

3. 已补用户侧的 `引导填写 / 直接填写` 说明
4. 已补填写说明弹框入口

当前待收口：

### 任务 4：虚拟机申请骨架

目标：

1. 先把最复杂也最常见的产品页骨架搭好
2. 先不追求全部字段和后端联动

首版区块：

1. 申请基础
2. 申请模式
3. 资源与部署
4. 组件能力选择
5. 配置详情
6. 补充信息

状态：待启动

当前说明：

1. 这是后续要优先落地的样板页
2. 重点不是先补内容，而是先把表单结构做出来
3. 需要先能承载环境、部署方式、组件选择和配置参考

建议下一步：

1. 先定虚拟机申请的页面骨架
2. 再定字段与配置参考的展示方式
3. 再把引导填写和直接填写两种模式接到同一套骨架上

### 任务 5：资源申请单继续收口

目标：

1. 明确它是“材料记录区”
2. 不与“我的工单”混淆

首版能力：

1. 草稿继续编辑
2. 已导出待评审记录回溯
3. 基于原单再发起
4. 删除
5. 筛选导出
6. 批量导出

状态：已完成首版

当前结论：

1. 资源申请单是“材料记录区”，不是正式工单
2. 支持草稿、导出、回溯、删除、筛选导出、批量导出
3. 后续如果重新编辑，建议作为新的申请发起处理

### 任务 6：完整服务目录降级处理

目标：

1. 保留原有能力
2. 不作为当前主路径

处理方式：

1. 改名为 `完整服务目录`
2. 从工作台和常见资源申请页提供弱入口
3. 页面文案明确“未覆盖场景再进入”

状态：已完成首版

当前已完成：

1. 页面名称已改为 `完整服务目录`
2. 首页与常见资源申请页已改为弱入口
3. 页面说明已明确“未覆盖场景再进入”

### 任务 7：填写说明入口收口

目标：

1. 不再作为一级菜单出现
2. 变成上下文辅助能力

处理方式：

1. 在常见资源申请页提供查看入口
2. 在引导填写 / 直接填写工作区提供查看入口
3. 使用弹框承载，不打断当前操作上下文

状态：已完成首版

当前结论：

1. `填写说明` 通过弹框承载，避免用户进出页面后找不到返回路径
2. 说明内容只保留在上下文里，不再单独成为主入口
3. 常见资源申请和填写工作区都可以唤起同一套说明内容

## 三、建议汇报顺序

建议按下面顺序和用户汇报：

1. 先看一级导航是否顺
2. 再看首页是否知道从哪里进入
3. 再看常见资源申请是否覆盖当前高频场景
4. 再看虚拟机申请骨架是否符合真实业务
5. 最后再逐步细抠字段和文案

---

## 四、当前风险

1. 参考平台字段较散，不能简单整页复制
2. Portal 当前材料整理字段不能被完全替换
3. 如果过早把 81 个原子服务抬到前台，会再次让用户无从下手
4. 如果现在就把正式工单入口做得过重，会干扰当前“先发起、先导出评审”的主目标

---
