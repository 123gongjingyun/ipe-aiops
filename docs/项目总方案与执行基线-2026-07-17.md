## 2. 项目定位

### 2.1 当前项目的真实定位

`project-mywayaiops-refactor-V2` 当前不是完整的正式交付平台，也不是完整的 ITSM 审批系统。

它当前更准确的定位是：

- 面向资源申请场景的服务门户与运营中心原型
- 聚焦“资源申请发起、材料整理、评审导出、正式工单跟踪、交付资产沉淀”
- 通过 Portal 承载申请人入口
- 通过 Center 承载运营和交付团队入口
- 通过 Shared 承载共享类型、配置、模板、规则和导出逻辑

### 2.2 后续目标定位

后续目标不是继续堆前端页面，而是把项目升级为：

- 可登录
- 可持久化存储
- 可多人在线使用
- 可基于历史单据复制和重新发起
- 可生成 HTML / PDF / Excel 评审材料
- 可在线维护字段字典、模板、套餐、布局和绑定关系
- 可逐步接入审批、ITSM、交付和验收全链路

也就是说，目标形态是：

**“资源申请与评审材料平台 + 正式工单跟踪 + 在线配置中心”**

## 3. 为什么现在要做架构升级

当前项目已经实现了较多页面和流程，但本质上仍偏前端原型，主要问题有：

1. 工单、模板、字段、套餐等运行时数据主要还在 `localStorage`
2. 当前只有开发态同步接口，没有正式后端
3. 没有正式登录态和菜单权限控制
4. 表单定义、布局定义、套餐绑定虽然已有雏形，但还没有做成在线配置维护后台
5. 导出能力目前偏 Excel 汇总，不是“评审材料体系”

如果继续在这个基础上只做页面功能叠加，会出现以下问题：

- 历史申请无法真正多人协同复用
- 数据不能稳定持久化
- 权限控制只能停留在前端模拟
- 字段一多、模板一多、流程一多，就会退回到“每次改代码”的低效模式

因此，当前阶段必须同步明确：

- 产品边界
- 技术架构
- 数据模型
- 配置中心路线
- 分阶段实施顺序

## 5. 当前表单实现机制

## 5.1 表单不是完全写死页面

当前 Portal 表单不是简单的手写固定页面，已经具有一定配置化能力。

表单运行链路大致如下：

1. 路由入口决定申请流类型
- [apply.tsx](/Users/gjy/project-mywayaiops-refactor-V2/src/packages/aiops-service-portal/src/pages/apply.tsx:1)

2. 页面组件负责编排申请流程
- [apply-wizard.tsx](/Users/gjy/project-mywayaiops-refactor-V2/src/packages/aiops-service-portal/src/components/apply-wizard.tsx:1)
- [apply-service.tsx](/Users/gjy/project-mywayaiops-refactor-V2/src/packages/aiops-service-portal/src/components/apply-service.tsx:1)
- `internet-app-wizard.tsx`

3. 字段和布局来自 Shared 配置
- `getResolvedSpecSchemaFields`
- `getResolvedSpecSchemaLayout`
- `getSchemaTemplate*`

4. 通用渲染器负责按配置渲染字段
- [template-form-layout.tsx](/Users/gjy/project-mywayaiops-refactor-V2/src/packages/aiops-service-portal/src/components/template-form-layout.tsx:1)

## 5.2 当前已经具备的配置化雏形

从数据结构看，当前已经有以下雏形：

- 字段字典
- 表单模板
- 模板版本
- 表单布局
- 套餐组 / 配置档位
- 模板与套餐绑定
- 服务与模板绑定

这说明当前方向不是错的，而是：

**已经做出“配置系统的基础内核”，但还没有把维护入口产品化。**

## 5.3 为什么 Center 里找不到同样的表单

因为当前共享的是：

- 模型
- 快照
- 模板定义
- 配置定义

而不是“Portal 的填写 UI”。

这也是合理的，因为：

- Portal 是填写入口
- Center 是运营和交付入口
- Center 当前应消费“快照”和“结果”，而不是承担同一份申请表单的填写职责

后续原则也应继续保持：

- 填写入口只在 Portal
- Center 不复制同一套填写 UI
- 历史复用仍然跳回 Portal 编辑

## 6. 目标产品形态

后续希望系统长成下面这个样子：

### 6.1 Portal

面向申请人，提供：

- 登录后访问
- 常见资源申请主入口
- 资源申请工作台
- 资源申请单记录与回溯
- 复制并编辑
- 重新发起
- HTML 评审版预览
- PDF / Excel 导出
- 有权限时进入完整服务目录
- 有权限时进入我的工单

### 6.2 Center

面向运营、交付、管理员，提供：

- 工单管理
- 交付详情
- 资产台账
- 服务规格治理
- 设置页
- 配置中心
- 角色与权限
- 后续审批与流程治理

### 6.3 配置中心

面向管理员和产品运营，提供：

- 字段字典管理
- 表单模板管理
- 模板版本管理
- 布局编辑器
- 套餐管理
- 模板与套餐绑定
- 服务与模板绑定
- 发布 / 回滚

### 6.4 导出体系

同一份申请数据可派生：

- `HTML 评审版`
- `PDF 打印版`
- `Excel 数据版`

## 9. 本轮新增与已确认需求

本轮经过讨论，已经明确的新增需求如下。

## 9.1 导出体系升级

1. 增加 `HTML` 模式
2. 增加 `PDF` 导出
3. 导出按填写表单审阅形式输出
4. 固定分两部分：
- `1. 用户需求`
- `2. 申请信息`
5. 非必填字段为空时也保留位置
6. 增加 `审批意见` 大框
7. 支持审批意见补录后再次导出
8. Excel 改成：
- `用户需求` 一个 sheet
- 每个组件一个 sheet

## 9.2 历史申请复用

1. 支持 `重新发起`
2. 支持 `复制并编辑`
3. 基于历史申请单直接回填
4. 目标是减少重复填写

## 9.3 Portal 首发权限收口

1. 首发默认不展示：
- `完整服务目录`
- `我的工单`

2. 保留菜单级权限能力
3. 首页和常见资源申请等说明性入口保留
4. 页面内无权限入口置灰并提示
5. 直接访问受限路由统一拦截

## 9.4 登录与权限

1. 引入登录登出
2. 引入当前用户信息
3. 菜单级权限控制
4. 后续支持用户级覆盖

## 12. 导出体系目标设计

## 12.1 导出体系不再只是 Excel

后续统一导出体系分为三层：

1. `HTML 评审版`
2. `PDF 打印版`
3. `Excel 数据版`

## 12.2 统一导出中间模型

三种导出形式不应分别维护独立逻辑，而应共用统一数据模型。

建议统一模型至少包括：

- 基本信息
- 用户需求区块
- 申请信息区块
- 组件区块
- 附件区块
- 评审概要概览
- 评审交流话术建议
- 审批意见区块
- 空字段占位说明

## 12.3 导出页面结构

导出页面结构固定为：

1. 用户需求
2. 申请信息
3. 评审概要概览
4. 评审交流话术建议
5. 审批意见

其中页面与导出的展示原则固定为：

1. `评审概要概览`
- Portal 填写页可以展示
- HTML / PDF / Excel 导出也展示
- 形式必须保持简短、结构化、适合快速扫读
2. `评审交流话术建议`
- 不放在日常填写页主界面
- 只放在 HTML / PDF / Excel 导出材料中
- 用于评审会前准备和现场交流
- HTML 评审页中默认折叠展示，避免正文过长影响审阅
- HTML 评审页应提供显式展开/收起入口，后续可增加 `复制话术` 能力
- PDF 导出保持可配置，默认建议不展开大段话术正文，按导出选项决定是否包含
3. `用户需求`
- 整块按必填区处理
- 除 `背景` 和 `网络需求-用户访问方式` 之外，其余允许在“不涉及”场景下明确填写 `无`
- `背景` 与 `用户访问方式` 不可填写 `无`
4. `重要总原则`
- 不挂在某个单字段下
- 作为 `用户需求` 分组顶部的强提示规则显示
- 原则是：必须明确数据规模、服务处理逻辑、数据库资源需求及其推算关系

Excel 结构固定为：

- 用户需求：单独 sheet
- 申请信息：单独 sheet
- 每个组件：单独 sheet
- 评审概要概览：单独 sheet
- 评审交流话术建议：单独 sheet
- 审批意见：单独 sheet

## 12.4 虚拟机申请页收口规则

`虚拟机申请` 页面在当前阶段补充收口如下：

1. 页面上只保留一个环境字段，统一命名为 `申请环境`
2. 旧的 `环境` 与 `申请环境` 不再双字段并存，避免用户重复理解和重复填写
3. `申请环境` 必须与 `申请模式` 保持联动
4. `申请环境 + 申请模式 + 虚拟机数量` 归并到同一组紧凑卡片中展示
5. `虚拟机数量` 不再独立漂浮在右侧，避免视觉断裂
6. 如内部已有旧字段存储结构，可通过映射方式兼容，但对用户只暴露一套口径

## 13. Portal 首发权限策略

## 13.1 首发默认策略

首发默认开放：

- 资源申请工作台
- 常见资源申请
- 资源申请单

首发默认关闭：

- 完整服务目录
- 我的工单

## 13.2 页面和导航处理原则

1. 主菜单无权限：
- 直接隐藏

2. 页面内辅助入口无权限：
- 保留认知性文案
- CTA 置灰
- 提示当前无权限

3. 路由直接访问无权限：
- 统一拦截

## 13.3 为什么不建议全隐藏

因为首页和常见资源申请页当前承担“边界说明”作用。

如果把能力名称完全隐藏，用户会误以为系统没有这些能力，而不是“当前账号未开通”。

因此推荐策略是：

- 认知层保留
- 操作层受控
