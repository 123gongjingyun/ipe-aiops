# Portal-菜单权限矩阵-2026-07-20

## 一、目的

本矩阵用于收口 `Portal` 侧登录后可见菜单的角色范围，作为后续：

1. 左侧导航渲染过滤
2. 页面级路由守卫
3. 用户登录后默认落地页判断

的统一依据。

本矩阵只定义 **菜单级与页面级访问权限**，不定义按钮级动作权限。

## 七、默认首页建议

为减少登录后跳转分歧，建议按角色给默认落地页：

| 角色 | 默认首页 |
| --- | --- |
| `applicant` | `/` |
| `delivery-engineer` | `/orders` |
| `reviewer` | `/orders` |
| `ops` | `/orders` |
| `security-admin` | `/orders` |
| `platform-admin` | `/` |

当用户具有多个角色时，建议按以下优先级选择默认首页：

1. `platform-admin`
2. `applicant`
3. `delivery-engineer`
4. `reviewer`
5. `ops`
6. `security-admin`

---

## 八、第一阶段实现建议

第一阶段只需要落下面这些能力：

1. `portal-layout` 按 `menuKey` 过滤导航
2. `App.tsx` 中为页面补 `RequireAuth`
3. 对非导航页面补 `RequireMenuAccess`
4. 登录成功后按默认首页规则跳转

当前不建议在第一阶段就做：

1. 动态菜单配置中心
2. 复杂的用户自定义菜单覆盖
3. 多租户 Portal 菜单差异

---

## 九、注意事项

1. `评审材料预览` 不应作为公开匿名页面保留
2. `资源申请单` 与 `评审材料预览` 在权限上应统一归到申请人材料视角
3. `完整服务目录` 可对交付侧开放，但不建议默认对所有内部角色开放所有编辑入口
4. Portal 阶段重点是“发起、整理、查看、导出”，不要把 Center 管理能力混入 Portal
