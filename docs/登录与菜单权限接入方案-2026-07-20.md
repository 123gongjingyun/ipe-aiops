# 登录与菜单权限接入方案-2026-07-20

## 一、目标

当前 `project-mywayaiops-refactor-V2` 已具备：

- Portal / Center 双端页面骨架
- Shared 公共层
- 角色定义种子与设置页中的角色维护能力

但仍缺少以下正式入口能力：

- 登录
- 登出
- 当前用户会话恢复
- 菜单按用户权限收口
- 页面级权限守卫

本方案目标是：在 **不重写当前业务页面结构** 的前提下，补齐统一认证层，并让 Portal / Center 共用一套登录态与权限判断逻辑。

## 四、设计原则

1. **不改现有业务字段**
   - 不影响已确认的申请表单字段、字段 key、导出字段与工单状态逻辑
2. **认证逻辑统一进 Shared**
   - Portal / Center 不各自复制一套认证状态机
3. **先做菜单与页面权限**
   - 第一阶段先收口入口与页面
4. **先 mock，后接后端**
   - 先跑通前端权限链路，再接真实接口
5. **复用逻辑，不复用旧 UI**
   - 借鉴 `mygetpre` 的认证模型，但不直接复制其页面壳与青绿色视觉

---

## 五、建议认证模型

建议新增统一认证用户结构：

```ts
interface AuthUser {
  id: string;
  username: string;
  displayName: string;
  email?: string;
  roleKeys: string[];
  roleLabels: string[];
  effectiveMenuKeys: string[];
  effectiveActionKeys: string[];
  isActive: boolean;
}
```

说明：

1. `roleKeys` 采用多角色数组，不采用单角色字符串
2. `effectiveMenuKeys` 作为菜单最终可见结果
3. `effectiveActionKeys` 作为动作最终可执行结果

---

## 六、建议文件结构

建议在 `shared` 中新增：

```text
src/packages/shared/src/auth/
├── auth-types.ts
├── auth-storage.ts
├── auth-permissions.ts
├── auth-api.ts
├── auth-context.tsx
├── require-auth.tsx
├── require-menu-access.tsx
├── login-page.tsx
└── mock-auth.ts
```

各文件职责如下：

### 6.1 `auth-types.ts`

定义：

- `AuthUser`
- `AuthSession`
- `MenuPermissionKey`
- `ActionPermissionKey`

### 6.2 `auth-storage.ts`

统一管理本地持久化：

- `getStoredAccessToken`
- `setStoredAccessToken`
- `clearStoredAccessToken`
- `getStoredCurrentUser`
- `setStoredCurrentUser`
- `clearStoredCurrentUser`

### 6.3 `auth-permissions.ts`

统一提供：

- `hasMenuAccess`
- `hasActionAccess`
- `hasAnyPermission`
- `buildEffectivePermissions`

### 6.4 `auth-api.ts`

统一封装：

- `login`
- `register`
- `logout`
- `fetchCurrentUser`

第一阶段可先走 mock，第二阶段再切真实接口。

### 6.5 `auth-context.tsx`

统一提供：

- `AuthProvider`
- `useAuth`

### 6.6 `require-auth.tsx`

未登录跳登录页。

### 6.7 `require-menu-access.tsx`

有登录但无页面菜单权限时跳首页或 `403`。

### 6.8 `login-page.tsx`

放共享登录页，Portal / Center 只传品牌配置，不各做一套页面。

## 十一、登录页视觉方案

### 11.1 视觉方向

登录页不复用旧项目青绿色调，统一改为当前平台丰田红风格。

建议色板：

- 主红：`#C8102E`
- 深红：`#9F1027`
- 浅红背景：`#FFF5F6`
- 深灰文案：`#1F2937`

### 11.2 设计原则

1. 白底卡片 + 红色主按钮
2. 激活态、焦点态、链接态统一红色
3. 背景可用浅红渐变，不做青绿科技风
4. Portal / Center 共用同一套登录页组件，只改品牌文案

---

## 十二、实施顺序

建议按下面顺序推进：

1. 先新增 Shared 认证层
2. 再接 Portal 登录与菜单过滤
3. 再接 Center 登录与菜单过滤
4. 再加页面级权限守卫
5. 最后切丰田红视觉

---

## 十三、验收口径

本阶段完成的最低验收标准：

1. 未登录访问 Portal / Center 自动跳登录页
2. 登录成功后进入对应首页
3. 刷新页面后仍能恢复登录态
4. 点击登出后清空会话并回登录页
5. 菜单按用户权限正确显示
6. 无权限页面不能靠地址栏直接进入

---

## 十四、风险与注意事项

1. 不要在第一阶段直接重构所有业务页面
2. 不要把权限逻辑散写进 Portal / Center 页面内部
3. 不要改变现有角色定义数据结构含义
4. 不要把登录 / 权限阶段和字段改造阶段混做
5. 注册能力默认保留实现入口，不强制开放 UI
