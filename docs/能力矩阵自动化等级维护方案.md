# 能力矩阵自动化等级维护方案

## 1. 背景

当前 `能力矩阵` 页面中的自动化等级 `手工 / 半自动 / 全自动`，主要依赖代码中的推导函数生成。

当前实现特点：

- 初始接入成本低
- 原型阶段可以快速产出矩阵结果
- 适合早期展示和讨论

但同时也存在明显问题：

- 规则偏粗糙，容易误判
- 口径被写死在代码里，不便调整
- 一旦服务实际能力变化，运营人员无法直接修正
- 页面展示值和真实交付能力之间可能逐步偏离

因此，更合理的方案应该是：

> 首次允许系统按规则自动判定；之后允许人工维护；并且判定规则本身也可以调整。

---

## 2. 目标

本方案的目标是建立一套兼顾“效率”和“准确性”的自动化等级维护机制。

### 2.1 需要满足的能力

1. 新服务创建时，系统能自动给出一个默认等级。
2. 运营人员可以手工修改某个服务的等级。
3. 后续再刷新、再发布时，不会把人工修改值覆盖掉。
4. 规则本身不是写死在代码里，而是可以维护和迭代。
5. 页面上可以区分当前值来自“系统判定”还是“人工确认”。

### 2.2 目标结果

最终能力矩阵中的自动化等级，不再只是“代码猜出来的值”，而是：

- 系统规则给出初始建议
- 人工可以校准
- 平台可持续演进规则

---

## 3. 建议的总体原则

建议把自动化等级的来源拆成两层：

### 3.1 第一层：初始判定

用于：

- 新建服务时自动给默认值
- 历史没有等级的服务做一次性补齐

特点：

- 允许粗粒度
- 允许规则迭代
- 目的是“提供起点”，不是最终事实

### 3.2 第二层：人工维护

用于：

- 修正系统误判
- 按实际能力口径调整
- 对关键服务给出正式结论

特点：

- 优先级高于系统推导
- 一旦人工确认，后续不应被自动覆盖
- 适合作为矩阵最终展示值

---

## 4. 推荐的优先级规则

建议最终展示值采用以下优先级：

### 4.1 展示优先级

1. 如果服务存在 `人工维护等级`，优先显示人工维护值
2. 如果没有人工维护值，则显示 `系统判定等级`
3. 如果系统判定也没有结果，则回退为 `manual`

可以表示为：

```text
最终展示值 = 人工维护值 ?? 系统判定值 ?? manual
```

### 4.2 规则解释

- “人工维护值”代表当前最可信口径
- “系统判定值”代表系统默认建议
- `manual` 是兜底值，不表示一定完全手工，只表示尚未可靠识别

---

## 5. 建议的数据模型

建议不要再只保留一个 `automationLevel` 字段，而是拆成三类信息。

### 5.1 推荐字段

#### 1. `automationLevelSuggested`

系统初始判定值。

示例：

- `manual`
- `semi`
- `full`

用途：

- 系统根据规则自动生成
- 可重新计算
- 不直接代表最终展示结论

#### 2. `automationLevelFinal`

最终展示值。

示例：

- `manual`
- `semi`
- `full`

用途：

- 用于能力矩阵展示
- 有人工维护时，直接写人工值
- 没人工维护时，可等于建议值

#### 3. `automationLevelSource`

记录当前最终值来源。

示例：

- `system`
- `manual`

用途：

- 页面上可视化区分“系统判定”或“人工确认”
- 后续做治理和审计时也更清楚

### 5.2 可选补充字段

#### 4. `automationLevelReason`

记录判定原因。

示例：

- `deliveryMode=ai`
- `关键词命中：配置/接入`
- `运营人工确认`

用途：

- 便于解释为什么系统给出这个等级
- 也便于回看人工修改依据

#### 5. `automationLevelUpdatedAt`

记录最后一次维护时间。

#### 6. `automationLevelUpdatedBy`

记录最后一次维护人。

---

## 6. 推荐的页面交互

### 6.1 服务目录编辑页

建议在原子服务编辑页增加“自动化等级”配置区域。

显示内容建议包括：

- 当前最终等级
- 来源：系统判定 / 人工维护
- 系统建议等级
- 判定原因
- 最近维护时间 / 维护人

可操作内容建议包括：

- 下拉选择：`手工 / 半自动 / 全自动`
- 按钮：`恢复系统建议`
- 文本框：`维护说明`

### 6.2 能力矩阵页

建议矩阵本身只展示“最终等级”，但可在详情页里补充来源信息。

例如：

- `半自动（人工确认）`
- `全自动（系统判定）`

或者用标识区分：

- `半自动`
  - 小字：`人工`
- `全自动`
  - 小字：`系统`

### 6.3 设置页

后续可以把规则维护收口到设置页，例如：

- 自动化等级判定规则
- 关键词维护
- `handler -> 等级` 映射
- `deliveryMode -> 等级` 映射

这样规则调整不需要每次改代码。

---

## 7. 初始判定规则建议

当前代码规则可以作为第一版的基础，但建议逐步升级。

### 7.1 第一版可接受规则

可以继续保留这些基础判断：

1. 显式 `automationLevel` 优先
2. `deliveryMode === 'ai'` 判 `full`
3. 命中关键词 `配置 / 接入 / 策略 / 开通` 判 `semi`
4. 其余判 `manual`

### 7.2 第二版建议加入更可靠依据

建议引入这些信号：

- `delivery.handler`
  - 是否存在真实自动执行器
- `delivery.dependencies`
  - 是否依赖人工前置动作
- 是否需要人工审批
- 是否涉及外部系统人工处理
- 是否只是生成方案、不真正执行

### 7.3 更合理的示例逻辑

例如：

- `deliveryMode === 'ai'` 且有真实执行 handler
  - `full`
- 有 handler，但关键步骤仍需人工确认或跨系统处理
  - `semi`
- 无 handler，纯人工执行
  - `manual`

这类规则比单纯依赖“关键词命中”更稳定。

---

## 8. 规则可配置的建议形式

### 8.1 推荐不要把规则继续硬编码

当前如果继续写死在 `deriveAutomationLevel()` 中，后续每次调整规则都要：

- 改代码
- 发版
- 回归

成本偏高。

### 8.2 建议的规则配置结构

可以用一个配置对象维护，例如：

```ts
{
  deliveryModeRules: {
    ai: "full",
    manual: null
  },
  keywordRules: [
    { keywords: ["配置", "接入", "策略", "开通"], level: "semi" }
  ],
  handlerRules: [
    { handlers: ["vm-provision", "db-provision"], level: "semi" }
  ],
  fallback: "manual"
}
```

后续再把这份规则：

- 放到本地配置
- 放到设置页
- 或放到后端配置表

都比较容易。

---

## 9. 推荐的最终流程

建议未来形成以下完整流程。

### 9.1 服务创建时

1. 用户新建原子服务
2. 系统根据规则自动生成 `automationLevelSuggested`
3. 默认把 `automationLevelFinal = automationLevelSuggested`
4. `automationLevelSource = system`

### 9.2 运营校准时

1. 运营人员在服务编辑页查看系统建议
2. 如有偏差，手工调整 `automationLevelFinal`
3. `automationLevelSource = manual`
4. 记录维护说明、时间、维护人

### 9.3 后续规则升级时

1. 系统可以重新批量计算 `automationLevelSuggested`
2. 对于 `source = system` 的服务，可选择同步刷新 `final`
3. 对于 `source = manual` 的服务，不自动覆盖

这样既能享受规则升级带来的收益，也不会破坏人工治理结果。

---

## 10. 推荐的系统行为

### 10.1 哪些情况允许自动覆盖

建议只在以下情况下允许自动覆盖最终值：

- 当前来源是 `system`
- 且用户从未手工维护过

### 10.2 哪些情况禁止自动覆盖

以下情况不应自动覆盖：

- 来源是 `manual`
- 用户手工维护过
- 有明确维护说明和责任人

### 10.3 恢复系统建议

建议提供“恢复系统建议”按钮，而不是强制覆盖。

这样人工维护仍然掌握主动权。

---

## 11. 当前阶段最合适的实施方式

考虑到当前项目仍处于原型和方案打磨阶段，建议按以下顺序落地。

### 第一阶段：低成本改造

目标：

- 保留当前初始推导能力
- 增加人工维护能力

建议动作：

1. 在原子服务数据模型中增加：
   - `automationLevelSuggested`
   - `automationLevelFinal`
   - `automationLevelSource`
2. 当前 `deriveAutomationLevel()` 输出作为 suggested
3. 能力矩阵展示改为优先读取 final
4. 服务目录编辑页允许人工修改 final

### 第二阶段：规则外置

目标：

- 判定规则不再写死在代码里

建议动作：

1. 抽出规则配置对象
2. 设置页增加规则维护能力
3. 支持批量重算 suggested

### 第三阶段：引入运行事实

目标：

- 把“能力定义”与“运行达成率”拆开

建议动作：

1. 保留 `automationLevelFinal` 作为能力标签
2. 另外新增“自动执行率、人工介入率、成功率”等运行指标
3. 在矩阵或详情页同时展示“定义能力”和“实际表现”

---

## 12. 推荐结论

建议后续统一采用以下原则：

1. 系统首次可以自动判定等级。
2. 自动判定只作为建议值，不应长期等同于最终值。
3. 最终展示值必须支持人工维护。
4. 人工维护值优先级高于系统规则。
5. 判定规则本身应逐步改为可配置，而不是长期写死在代码里。

这样既保留了原型阶段“快速生成结果”的效率，又能逐步过渡到更稳定、更可信的运营口径。
