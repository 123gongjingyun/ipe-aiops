# 资源申请推荐套餐规则草案

> 创建日期：2026-07-01  
> 适用范围：`project-mywayaiops-refactor-V1` Portal 发起页 / Shared 推荐逻辑 / Center 评审  
> 目标：明确资源申请发起后，系统应如何根据主输入字段给出容器、虚机、数据库与平台接入的推荐套餐，并定义非标判定边界

---

## 一、适用边界

本草案只定义：

- 输入字段如何影响推荐
- 系统推荐哪些标准套餐
- 什么情况下标记为非标
- 什么情况下只给建议、不直接生成推荐

本草案不定义：

- 当前工单主流程状态推进
- ITSM 审批节点变化
- 交付结果回填格式

当前主流程状态机保持不变：

`pending -> reviewing -> processing -> plan_confirming -> delivering -> completed -> confirmed -> archived`

---

## 二、推荐逻辑总原则

推荐套餐的目标不是“替用户做最终资源设计”，而是先给出一版标准初稿，降低 Excel 手填成本。

建议遵循下面 5 条原则：

1. 优先用结构化字段给推荐，不依赖自由文本猜测全部资源
2. 先推标准套餐，再允许有限调整
3. 超出边界即标记为非标，而不是继续假装标准
4. 缺关键输入时，只给缺失提示，不输出伪精确推荐
5. 推荐结果只影响表单初稿和评审关注点，不直接改变主流程状态

---

## 三、输入字段分层

推荐逻辑建议优先消费下面几类输入。

### 3.1 一级输入：决定推荐主方向

| 字段key | 作用 |
|---|---|
| `environment` | 决定 DEV / SIT / UAT / PROD 资源规格与保障级别 |
| `businessCategory` | 决定业务域基础场景 |
| `businessCriticality` | 决定是否偏保守配置 |
| `scale` | 决定小/中/大资源档位 |
| `applicationDescription` | 用于辅助判断应用形态 |
| `businessPurpose` | 用于辅助判断用途和基础套餐 |
| `publicAccess` | 决定是否生成公网/域名/负载均衡相关建议 |
| `requiresInterconnect` | 决定是否生成网络互访建议 |

### 3.2 二级输入：细化推荐规格

| 字段key | 作用 |
|---|---|
| `userScale` | 细化容量级别 |
| `concurrencyLevel` | 细化容量级别 |
| `haRequirement` | 决定单实例、主备、多AZ、双活倾向 |
| `requiresBackup` | 决定备份组件与策略建议 |
| `backupPolicy` | 决定备份默认方案 |
| `rtoTarget` | 约束 HA / 容灾保守程度 |
| `rpoTarget` | 约束备份与复制策略 |
| `drRequirement` | 决定是否升级为跨可用区或异地容灾 |
| `integrationSystems` | 辅助判断是否需要网关、互访、消息、缓存等 |
| `clientTypes` | 辅助判断前端/网关/开放接口场景 |

### 3.3 三级输入：推荐后的用户调整

| 字段key | 作用 |
|---|---|
| `instanceCount` | 调整实例数 |
| `cpu` | 调整 CPU |
| `memory` | 调整内存 |
| `systemDiskSize` | 调整系统盘 |
| `dataDiskSize` | 调整数据盘 |
| `resourcePurposeDetail` | 补充特殊用途 |

---

## 四、标准推荐场景分类

建议先把资源推荐收敛为 5 类主场景：

1. 通用应用容器场景
2. 通用应用虚机场景
3. 数据库场景
4. 中间件场景
5. 外网/开放访问场景

系统可以根据业务输入命中一到多个场景，再组合输出推荐清单。

---

## 五、容器套餐推荐规则

## 5.1 适用判断

优先推荐容器套餐的典型条件：

- `applicationDescription` / `businessPurpose` 指向 Web、接口、平台、门户、服务类应用
- `clientTypes` 包含 `Web` / `H5` / `API`
- 未显式声明必须上虚机或特殊中间件
- 架构图中呈现应用层、网关层、服务层结构

### 5.2 标准容器套餐建议

#### 小型

适用：

- `environment` 为 `DEV` / `SIT`
- `scale=small`
- `businessCriticality=standard`

推荐：

- 后端容器：`1C / 2G / 1~2 实例`
- 前端容器：`1C / 1G / 1~2 实例`
- 默认接入日志与监控

#### 中型

适用：

- `environment=UAT`
- 或 `scale=medium`

推荐：

- 后端容器：`2C / 4G / 2 实例`
- 前端容器：`1C / 2G / 2 实例`
- 默认日志、监控、基础备份策略

#### 大型 / 生产标准

适用：

- `environment=PROD`
- 或 `businessCriticality` 为 `important / critical`

推荐：

- 后端容器：`4C / 8G / 2~3 实例`
- 前端容器：`2C / 4G / 2 实例`
- 建议多实例部署
- 默认开启监控、日志、备份建议

### 5.3 容器套餐非标判定

命中任一条件即建议标记非标：

- 用户把 `instanceCount` 调整到标准上限以上
- 单实例 `cpu` 或 `memory` 超过标准档位两级以上
- `drRequirement` 要求异地多活
- `publicAccess=true` 且还要求复杂跨系统访问 / 多域名 / 特殊安全控制
- 架构图体现复杂多层服务编排，超出标准应用模板

---

## 六、虚机套餐推荐规则

## 6.1 适用判断

优先推荐虚机套餐的典型条件：

- `resourcePurpose` 明确为应用主机 / 工具主机 / 特殊运行环境
- `applicationDescription` 指向传统部署、代理、文件服务、特殊环境依赖
- 架构图体现单机或少量节点运行
- 不适合容器化或需保留传统中间件方式

### 6.2 标准虚机套餐建议

#### 基础虚机

适用：

- `environment=DEV/SIT`
- 非关键业务

推荐：

- `2C / 4G / 系统盘 50G / 数据盘 100G`

#### 通用业务虚机

适用：

- `environment=UAT`
- 普通业务应用

推荐：

- `4C / 8G / 系统盘 100G / 数据盘 200G`

#### 生产业务虚机

适用：

- `environment=PROD`
- 核心或重要业务

推荐：

- `8C / 16G / 系统盘 100G / 数据盘 300~500G`
- 默认建议双节点或主备

### 6.3 虚机套餐非标判定

- 单机规格超出 `16C / 32G`
- 要求 4 台以上同规格节点且无明确标准模板
- 既要求生产高可用又要求复杂公网 / 复杂互访
- 特殊操作系统 / 特殊镜像要求

---

## 七、数据库套餐推荐规则

## 7.1 适用判断

优先推荐数据库套餐的条件：

- `resourcePurpose=database`
- `databaseName` 已给出或能推断
- 架构图有数据库节点
- `applicationDescription` 出现订单、交易、报表、主数据、缓存等数据持久化特征

### 7.2 标准数据库套餐建议

#### 开发/测试数据库

适用：

- `environment=DEV/SIT`

推荐：

- `2C / 4G / 100~200G`
- 默认单实例
- 默认标准备份

#### UAT 数据库

适用：

- `environment=UAT`

推荐：

- `4C / 8G / 200~500G`
- 可建议主备或至少具备恢复方案

#### 生产数据库

适用：

- `environment=PROD`

推荐：

- `8C / 16G / 500G 起`
- 默认主备或高可用模式
- 默认推荐备份、监控、日志接入

### 7.3 数据库补充建议

系统可附带推荐：

- `charset` 默认值
- `backupPolicy` 默认值
- `retentionPeriod` 默认值
- `haRequirement` 默认值

### 7.4 数据库非标判定

- `dataDiskSize` 超过标准上限
- 要求双活、多主、跨地域灾备
- 要求特殊数据库版本或特殊字符集组合
- 高并发、高写入、低 RPO/RTO 组合超出标准模板

---

## 八、中间件套餐推荐规则

## 8.1 适用判断

适用于：

- `resourcePurpose` 为缓存、消息、检索、日志、网关等
- `integrationSystems` 较多，需要解耦或异步处理
- 架构图体现 Redis / MQ / Kafka / ES 等组件

### 8.2 标准中间件建议

#### Redis / 缓存

推荐逻辑：

- DEV/SIT：单实例或基础主从
- UAT：小集群
- PROD：高可用集群或哨兵模式

#### 消息队列

推荐逻辑：

- 有系统互访和异步场景时才推荐
- UAT 以上默认建议至少 2~3 节点

#### 日志 / 检索

推荐逻辑：

- 默认不在首轮强推独立集群
- 仅当架构图或需求明确时推荐独立日志/检索资源

### 8.3 中间件非标判定

- 用户要求大量组件混合但无明确边界
- 需要跨区域同步或超大集群
- 性能和可用性要求明显超出标准模板

---

## 九、外网与开放访问推荐规则

## 9.1 适用判断

当下列条件成立时，系统应补充外网访问类推荐：

- `publicAccess=true`
- `domainName` 有值
- `listenerProtocol` / `listenerPort` 有值

### 9.2 推荐内容

建议推荐：

- 负载均衡 / 网关
- 域名与证书
- 安全审查提示
- 日志与监控增强

### 9.3 非标判定

- 多域名、多协议、多端口组合
- 复杂跨网区链路
- 特殊安全合规要求

---

## 十、平台接入建议规则

这部分不是“资源规格”，但应一并作为推荐结果给出。

### 10.1 默认建议

| 条件 | 建议项 |
|---|---|
| 容器/PaaS 场景 | `requiresPipelineAccess=true` |
| 有日志排查需求 | `requiresEfkAccess=true` |
| 正常业务系统 | `requiresMonitoring=true` |
| 正常业务系统 | `requiresLogging=true` |

### 10.2 非标判定

一般不单独作为非标，但如果用户显式关闭监控/日志且又是 `PROD + critical`，应在评审中高亮风险。

---

## 十一、推荐失败或降级策略

当缺少关键输入时，系统不应强行输出看似准确的套餐。

### 11.1 仅给提示、不生成具体套餐

命中以下情况时，只提示待补信息：

- 缺 `environment`
- 缺 `businessPurpose` 和 `applicationDescription`
- 架构图缺失且业务描述过于抽象
- 输入内容同时指向多个资源方向但无优先级

### 11.2 只给方向，不给精确规格

命中以下情况时，系统只建议资源类别：

- 只知道要“上数据库”，不知道容量和环境
- 只知道要“开外网”，但不清楚访问模式
- 只知道要“部署应用”，但无法判断容器还是虚机

---

## 十二、非标标记规则

建议系统统一产出一个非标判断结果，而不是各模块各自判断。

### 12.1 建议输出结构

```ts
recommendationAssessment?: {
  standard: boolean
  reasonCodes: string[]
  reasons: string[]
}
```

### 12.2 典型 reasonCodes

| code | 含义 |
|---|---|
| `oversized_spec` | 规格超标准档位 |
| `complex_network` | 网络关系复杂 |
| `strict_dr` | 容灾要求过高 |
| `special_os` | 特殊操作系统/镜像 |
| `unknown_capacity` | 容量信息不足 |
| `multi_component_custom` | 多组件组合场景复杂 |

### 12.3 非标后的处理

非标标记只影响：

- 页面提示
- 评审关注点
- 后续架构/网络/安全评审重点

非标标记默认不直接影响：

- 工单状态机推进
- 阶段节点显示逻辑

---

## 十三、和数据模型的关系

推荐套餐规则建议和字段数据模型按下面方式衔接：

- 输入字段：来自 `order.answers`
- 辅助标记：来自 `order.extras` 和附件元数据
- 推荐结果：落 `orchestratedPlan.resources`
- 非标结论：可放推荐评估结构或 `extras`
- 评审修正：落评审补充结构

这样可以保持：

- 用户输入
- 系统推荐
- 评审修正
- 交付结果

四层解耦。

---

## 十四、建议下一步

如果继续推进，下一步建议补一份：

`推荐结果映射草案`

重点回答：

- 推荐出的容器/虚机/数据库/中间件结果，如何映射到 `orchestratedPlan.resources`
- 哪些推荐字段同时需要展示在发起页
- 哪些推荐字段只进入评审和导出，不在发起页直接露出

这样就可以从“规则草案”进入“真实数据结构映射”。
