## 八、近期执行建议（未来 2 周）

按优先级排序，参考改造方案分阶段实现：

### Phase 1：数据层扩展（1.5 天）

1. **P1-2 字段字典数据层：FieldDictionaryEntry + ServiceFieldBinding 类型定义**
   - 新建 `shared/data/field-dictionary.ts`：FieldDictionaryEntry 接口 + 种子数据（description/urgency/deadline 等通用字段）
   - 新建 `shared/store/field-dictionary.ts`：字段字典 Store（CRUD、引用计数）
   - 扩展 `shared/types/index.ts`：新增 FieldDictionaryEntry、ServiceFieldBinding 接口

2. **P1-3 配置数据层：ServiceConfigProfile + ComboConfigTemplate 类型定义**
   - 新建 `shared/data/config-profiles.ts`：ServiceConfigProfile 接口 + 仿生产环境种子数据（30 条核心）
   - 新建 `shared/data/config-templates.ts`：ComboConfigTemplate 接口
   - 扩展 `shared/store/orders.ts`：订单结构新增 applicantId/env/slaLevel/selectedProfileId/approvalChain 等字段
   - 扩展 `shared/store/specs.ts`：spec 对象新增 configProfiles 数组 + inputSchema 支持字段绑定

3. **P1-10 需求细化单附件与 AI 分析结果模型**
   - 扩展订单/草稿结构：新增 `attachments`、`architectureAssets`、`importedConfigDocs`、`aiAnalysisSummary`
   - 统一上传材料元数据：文件类型、来源、解析状态、关联版本
   - 为后续导出 Excel / PDF 评审材料预留数据结构
### Phase 2：Portal 发起页增强项预留（后续）

4. **P1-10 发起页保持轻量，材料上传作为增强项**

## 九、关键设计决策（已确认）

| 决策项 | 决策 | 与仿生产环境的关系 |
|--------|------|------------------|
| **字段数据字典** | ✅ 放入「设置」页面作为标签页 | 解决 60+ 服务字段重复定义问题；与仿生产环境「配置选项」管理能力对齐；变更仅影响新绑定 |
| **配置数据字典（ServiceConfigProfile）** | ❌ 否，配置依附于服务编辑页 | 仿生产环境有独立配置管理页面；配置与服务强相关，全局独立不合理 |
| **配置文档模板** | ✅ 保留，但降级为导入/导出与参考能力 | 不再作为首轮发起主入口，避免把线下 Excel 流程线上化 |
| **架构图上传** | ✅ 提升为 Portal 发起页主输入之一 | 更贴近真实环境需求，适合作为 AI 推荐和校验的主要依据 |
| **字段字典变更同步策略** | 弱一致：变更只影响新绑定，已有绑定保留原值 | 保障服务自治性，避免字典一改导致全局服务异常 |
| **服务编辑页面** | 复用现有「全部服务」编辑弹窗，输入输出从字段字典选择绑定 + 配置数据字典折叠面板 | 与仿生产环境配置 CRUD 能力对齐，字段级独立管理 + 配置级依附服务 |
| **组合模板管理** | 复用现有组合服务编辑弹窗，在其内增加推荐配置模板区域 | 仿生产环境无组合服务概念，本方案新增，并用于支撑架构图识别后的自动推荐 |
| **SLA 管理** | ✅ 完全保留现有 SLA 管理页面 | 仿生产环境无 SLA 概念，本方案保留并扩展（SLA 与配置适配） |
| **角色/用户管理位置** | 放到「设置」页面下作为二级标签 | 仿生产环境有独立角色/用户管理；本方案收敛到设置页，保持导航简洁 |
| **审批流程配置** | 独立页面，支持可视化流程图 | 与仿生产环境审批能力对齐，本方案增强为可视化编排 |
| **批量申请** | ⏳ 本期不做 | 仿生产环境已支持；本方案留二期 |
