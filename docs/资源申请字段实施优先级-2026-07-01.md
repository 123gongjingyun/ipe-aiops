# 资源申请字段实施优先级

> 创建日期：2026-07-01  
> 适用范围：`project-mywayaiops-refactor-V1` Portal / Center / Shared  
> 目标：在不影响当前主流程状态判断的前提下，给资源申请字段收口提供可执行排期顺序

---

## 一、前提约束

本清单只用于指导“字段与页面承载”的实施顺序，不改变当前工单主流程状态机：

`pending -> reviewing -> processing -> plan_confirming -> delivering -> completed -> confirmed -> archived`

执行约束：

1. 新增字段默认先挂到 `answers` / `extras` / 附件元数据
2. 不把新增字段直接写进流程状态推进条件
3. 评审和交付补充字段与主流程解耦
4. 涉及详情页阶段展示、时间线、阶段状态的改动，必须单独回归

---

## 二、实施目标拆分

这轮资源申请字段收口，建议拆成 4 个阶段：

1. `阶段 A`：发起页主字段补齐
2. `阶段 B`：推荐套餐与可调字段补齐
3. `阶段 C`：评审补充字段与附件规范补齐
4. `阶段 D`：交付/运营回填与导出对齐

排序逻辑：

- 先让用户把需求说清楚
- 再让系统能推荐资源
- 再让评审和交付有落点
- 最后再考虑报表和历史模板对齐

---

## 三、P0：本轮最先补齐

这批字段优先级最高，因为它们最直接决定“能不能替代 Excel 作为发起入口”。

### 3.1 发起页基础缺口

| 字段key | 中文名称 | 作用 | 原因 |
|---|---|---|---|
| `integrationSystems` | 对接系统列表 | 表达系统联携关系 | 对应用户需求表核心信息 |
| `userTypes` | 用户类型 | 区分内部/外部/经销商等使用人群 | 影响网络、安全、容量判断 |
| `clientTypes` | 访问终端类型 | PC / H5 / App / API 等 | 影响架构和访问方案 |
| `applicationDescription` | 应用描述 | 发起主说明 | 虽已有定义，但必须作为必填主项稳定保留 |
| `businessPurpose` | 业务用途 | 发起主说明 | 对推荐和评审最关键 |

### 3.2 资源推荐基础缺口

| 字段key | 中文名称 | 作用 | 原因 |
|---|---|---|---|
| `resourcePartition` | 资源分区 | 容器/虚机推荐归属 | 对应资源表核心字段 |
| `resourcePurpose` | 资源用途 | 虚机/数据库/中间件用途 | 影响套餐和审批判断 |
| `databaseName` | 数据库名 | 数据库类资源标识 | Excel 虚机表核心字段 |
| `charset` | 字符集/编码 | 数据库类资源补充 | 运维落地需要 |
| `logRetentionDays` | 日志保留天数 | 运维策略默认值 | Excel 虚机表核心字段 |

### 3.3 SLA/备份基础缺口

| 字段key | 中文名称 | 作用 | 原因 |
|---|---|---|---|
| `requiresBackup` | 是否需要备份 | 开启备份策略分支 | 不能只靠备注表达 |
| `restoreRequirement` | 恢复要求 | 约束备份恢复能力 | SLA 核心输入 |
| `rtoTarget` | RTO目标 | 恢复时效目标 | SLA 核心输入 |
| `rpoTarget` | RPO目标 | 数据恢复点目标 | SLA 核心输入 |
| `drRequirement` | 容灾要求 | 是否主备/跨AZ/异地 | SLA 核心输入 |

### 3.4 附件与平台权限基础缺口

| 字段key | 中文名称 | 作用 | 原因 |
|---|---|---|---|
| `architectureDiagram` | 架构图附件 | 替代线下架构图模板 | 主路径要件 |
| `requiresPipelineAccess` | 是否需要流水线权限 | 容器/PaaS 典型需求 | 租户表核心信息 |
| `requiresEfkAccess` | 是否需要EFK权限 | 日志查看权限 | 租户表核心信息 |

**P0 目标**

- 先让发起页能完整承接用户需求表、SLA关键项、资源申请关键项
- 先不要追求把所有 Excel 列一次性搬完
- 先保证“发起信息 -> 推荐套餐 -> 评审理解”这条链路能闭合

## 六、建议实施顺序

### 6.1 第一轮

只做 P0：

- 补发起页缺口字段
- 补架构图附件承载
- 补推荐套餐关键字段
- 不改主流程判断
- 不做复杂导出改造

### 6.2 第二轮

补 P1：

- 加强网络、容量、监控和图完整性校验
- 细化推荐规则和非标标记
- 仍尽量不触碰状态判断逻辑

### 6.3 第三轮

补 P2：

- 把评审补充、交付回填、商流字段补齐
- 再看是否需要同步导出和后台统计口径

---

## 七、实施时的技术边界

后续研发落地时，建议遵守下面边界。

### 7.1 可以做的

- 扩展 `shared` 字段定义和页面字段分组
- 扩展发起页表单和推荐视图
- 扩展附件元数据
- 扩展评审补充表单
- 扩展交付回填结构

### 7.2 暂时不要直接做的

- 不要把新增字段直接塞进 `workflowTimeline` 状态推进判断
- 不要为了导出表头对齐 Excel，就修改 Portal / Center 当前阶段节点逻辑
- 不要在 `portal/order-detail.tsx`、`center/order-detail.tsx` 里重新各写一套状态判断

### 7.3 如必须改流程判断

仅在满足下面条件时才考虑：

1. 字段确实是流程门禁条件，而非展示或补充信息
2. 已明确 shared 层收口方案
3. 已补对应单元测试和页面回归测试

---

## 八、推荐的下一步产出

如果后续继续推进，建议按顺序产出：

1. 发起页字段分组原型表
2. 字段数据模型草案
3. 推荐套餐规则草案
4. 评审节点补充字段草案
5. 导出字段映射表

这样能先把需求和模型对齐，再进入 UI 和实现阶段。
