## 三、总体结论

参考平台并不是一套统一字段的大表单，而是：

1. 以产品页为中心
2. 每个产品有自己的独立字段
3. 共享的是页面骨架和申请管理方式，而不是完全相同字段

因此当前 Portal 的“统一”不应理解成：

- 所有申请页共用完全相同表单字段

而应理解成：

- 共用统一产品定义框架
- 每个产品各自维护自己的字段 schema

## 四、产品级差异摘要

### 4.1 虚拟机申请

参考平台字段：

- 系统编号
- 系统名称
- 模块名称
- 担当
- 环境
- 类型
- 配置选择
- 系统盘
- 数据盘

Portal 当前已有抽象：

- 系统名称
- 应用名称
- 环境
- 使用对象
- 应用形态
- 访问终端
- 业务目标
- 对接系统
- 访问与网络
- 资源诉求
- SLA / 安全 / 容灾
- 架构材料说明

差异结论：

1. 参考平台更偏资源配置与业务标识
2. Portal 当前更偏“申请意图结构化”
3. 两者不能简单二选一

建议：

- 保留 Portal 的申请意图结构
- 吸收参考平台的业务识别字段
- 暂不完全退回旧平台的 `类型 + 配置选择` 表达法

### 4.2 容器申请

参考平台字段：

- 环境
- 系统代码
- 供应商
- 应用英文名
- 应用描述
- 实例数
- 单 CPU
- 单内存
- 备注
- 申请人

差异结论：

1. 容器申请非常资源量化
2. 与虚拟机申请的结构化意图字段体系不一致

建议：

- 独立产品 schema
- 不并入通用资源申请大表单

### 4.3 OBS

参考平台字段：

- 环境
- 申请人
- 域账号
- 应用供应商名称
- 所属系统
- 应用担当
- 应用所属业务
- 桶名称
- 桶内目录名称
- 容量大小
- AK/SK 数量
- 使用时间
- 备注

建议：

- 作为独立产品 schema
- 与 SFS 抽共享基础骨架

### 4.4 SFS

参考平台字段：

- 环境
- 申请人
- 域账号
- 应用供应商名称
- 所属系统
- 应用担当
- 应用所属业务
- SFS 名称
- 容量大小
- 使用时间
- 备注

建议：

- 与 OBS 共用基础字段
- 仅保留特有对象字段差异

### 4.5 用户权限申请

参考平台字段：

- 域账号
- 姓名
- 手机号码
- 邮箱
- 申请权限（多选）

建议：

- 独立产品页
- 不强并入资源类工作台字段

### 4.6 网络策略申请

参考平台字段：

- 环境
- 源资产编号
- 源地址
- 目标资产
- 目标地址
- 所属系统
- 端口类型
- 端口

建议：

- 独立产品页
- 与资源申请工作台入口统一，但字段独立

## 五、当前阶段替换原则

### 5.1 先新增，不先硬替换

当前阶段建议：

1. 先做新页面结构
2. 先引入产品化入口
3. 先把字段清单明确
4. 再逐步决定哪些字段进入 Portal 主模型

### 5.2 暂不直接替换 Portal 现有字段的部分

以下内容当前建议先保留 Portal 自己的结构：

- 业务目标
- 对接系统
- 访问与网络
- 资源诉求
- SLA / 安全 / 容灾
- 架构材料说明

原因：

这些字段更贴近当前“评审材料整理与导出”的目标，不宜直接被旧平台资源配置字段替代。

### 5.3 建议优先吸收的字段

优先建议从参考平台吸收：

- 系统编号 / 系统代码
- 模块名称
- 担当 / 应用担当
- 应用供应商名称
- 应用所属业务
- 容量大小
- 使用时间
- 端口类型 / 端口
- 权限集合
