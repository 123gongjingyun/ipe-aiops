# 资源申请推荐结果映射草案

> 创建日期：2026-07-01  
> 适用范围：`project-mywayaiops-refactor-V1` Shared 推荐编排 / Portal 发起页 / Center 评审页  
> 目标：明确“发起页输入 -> 推荐套餐 -> `orchestratedPlan.resources` / 页面展示 / 评审补充”的映射关系，避免推荐逻辑与数据结构脱节

---

## 一、适用边界

本草案回答 3 个问题：

1. 推荐出来的资源结果如何落到 `orchestratedPlan.resources`
2. 哪些推荐字段要直接在发起页展示
3. 哪些推荐字段只用于评审、导出或提示，不进入资源清单

本草案不改变：

- 当前工单主流程状态机
- Portal / Center 阶段节点判断
- ITSM 审批流

当前主流程状态机保持：

`pending -> reviewing -> processing -> plan_confirming -> delivering -> completed -> confirmed -> archived`

---

## 二、推荐结果的三层输出

建议系统推荐始终输出三层结果，而不是只有一份资源数组。

### 2.1 页面展示层

用于发起页和评审页快速阅读：

- 推荐资源卡片
- 推荐原因
- 非标提示
- 缺失项提示

### 2.2 编排资源层

用于后续交付和导出：

- `orchestratedPlan.resources`

### 2.3 评估结论层

用于评审和风险提示：

- `recommendationAssessment`
- `missingInputs`
- `adviceNotes`

建议不要把这三层混成一个对象。

---

## 三、建议的数据结构

在不破坏现有结构的前提下，建议推荐结果按下面方式组织：

```ts
orchestratedPlan: {
  summary: string
  estimatedTime: string
  resources: ResourceRecommendation[]
  integrations?: IntegrationRecommendation[]
  recommendationAssessment?: {
    standard: boolean
    reasonCodes: string[]
    reasons: string[]
  }
  missingInputs?: string[]
  adviceNotes?: string[]
}
```

其中：

- `resources` 是真正进入编排和交付的资源清单
- `recommendationAssessment` 用于标记是否标准套餐
- `missingInputs` 用于提示还缺什么信息
- `adviceNotes` 用于补充说明，不直接驱动流程状态

---

## 四、输入到推荐结果的映射主线

建议按下面主线理解：

`order.answers`  
-> 识别资源场景  
-> 生成推荐资源项  
-> 生成页面展示文案  
-> 生成非标评估与缺失项  
-> 写入 `orchestratedPlan`

其中：

- 申请输入是源头
- 推荐结果是中间层
- 交付结果是后续阶段独立生成物

不能把最终交付资产直接反向当推荐结果。

---

## 五、容器推荐结果映射

## 5.1 建议映射到 `orchestratedPlan.resources`

当系统判定为容器场景时，建议每类应用生成一条或多条资源记录。

### 建议资源对象

```ts
{
  type: 'paas',
  name: '订单中心后端容器组',
  spec: {
    cpu: '2C',
    memory: '4G',
    instances: 2,
    resourcePartition: 'LAN'
  },
  purpose: '承载订单中心后端服务',
}
```

### 字段映射建议

| 输入字段 | 推荐结果字段 | 说明 |
|---|---|---|
| `applicationName` | `name` | 拼装资源名称 |
| `resourcePartition` | `spec.resourcePartition` | 资源分区 |
| `cpu` | `spec.cpu` | 推荐或用户调整后的 CPU |
| `memory` | `spec.memory` | 推荐或用户调整后的内存 |
| `instanceCount` | `spec.instances` | 推荐或用户调整后的实例数 |
| `resourcePurpose` / 场景识别 | `purpose` | 资源用途 |

### 页面展示建议

发起页展示：

- 资源名称
- 资源类型：`PaaS / 容器`
- 规格：CPU / 内存 / 实例数
- 推荐原因
- 是否标准套餐

### 不建议直接进资源对象的字段

下面字段不建议直接塞进 `resources.spec`：

- `requiresPipelineAccess`
- `requiresEfkAccess`
- `requiresMonitoring`
- `requiresLogging`

这些更适合放在平台接入建议或附属能力建议里。

---

## 六、虚机推荐结果映射

## 6.1 建议映射到 `orchestratedPlan.resources`

当系统判定为虚机场景时，建议每个用途生成一条资源记录。

### 建议资源对象

```ts
{
  type: 'vm',
  name: '报表服务虚机',
  spec: {
    cpu: '4C',
    memory: '8G',
    systemDisk: '100G',
    dataDisk: '200G',
    os: 'CentOS 7.9'
  },
  purpose: '承载报表应用服务'
}
```

### 字段映射建议

| 输入字段 | 推荐结果字段 | 说明 |
|---|---|---|
| `applicationName` | `name` | 组合资源名称 |
| `resourcePurpose` | `purpose` | 虚机用途 |
| `cpu` | `spec.cpu` | 规格 |
| `memory` | `spec.memory` | 规格 |
| `systemDiskSize` | `spec.systemDisk` | 规格 |
| `dataDiskSize` | `spec.dataDisk` | 规格 |
| `osImage` | `spec.os` | 镜像/操作系统 |

### 页面展示建议

发起页展示：

- 资源名称
- 规格摘要
- 适用原因
- 是否标准套餐

### 仅评审/导出保留，不进入主资源对象的字段

- `logRetentionDays`
- `charset`

这两个字段更接近运维与数据库配置，建议保留在申请字段或评审补充中，不必强行塞进所有 `vm` 资源对象。

---

## 七、数据库推荐结果映射

## 7.1 建议映射到 `orchestratedPlan.resources`

数据库推荐应尽量单独生成资源项，不和应用主机混合。

### 建议资源对象

```ts
{
  type: 'db',
  name: '订单中心 MySQL 实例',
  spec: {
    cpu: '4C',
    memory: '8G',
    storage: '500G',
    engine: 'MySQL 8.0',
    ha: '主备'
  },
  purpose: '承载订单中心核心业务数据库'
}
```

### 字段映射建议

| 输入字段 | 推荐结果字段 | 说明 |
|---|---|---|
| `databaseName` | `name` / 后续交付命名参考 | 可作为资源命名基础 |
| `cpu` | `spec.cpu` | 推荐数据库规格 |
| `memory` | `spec.memory` | 推荐数据库规格 |
| `dataDiskSize` | `spec.storage` | 存储容量 |
| `charset` | 保留在申请字段/评审建议 | 不强塞进 `spec` 也可 |
| `haRequirement` | `spec.ha` | 高可用模式 |

### 页面展示建议

发起页展示：

- 数据库引擎
- 规格
- 是否高可用
- 备份建议

### 额外建议

`backupPolicy`、`retentionPeriod`、`rtoTarget`、`rpoTarget` 更适合作为资源卡片旁边的“保障策略摘要”，不一定写进 `db.spec`。

---

## 八、中间件推荐结果映射

## 8.1 建议映射到 `orchestratedPlan.resources`

Redis、MQ、Kafka、日志检索类中间件建议独立成资源项。

### 建议资源对象

```ts
{
  type: 'middleware',
  name: '订单中心 Redis 集群',
  spec: {
    topology: '3主3从',
    memory: '16G',
    version: 'Redis 6'
  },
  purpose: '承载缓存能力'
}
```

### 字段映射建议

| 输入字段/识别结果 | 推荐结果字段 | 说明 |
|---|---|---|
| `resourcePurpose=cache` 或场景识别为缓存 | `type=middleware` + `name` | 中间件类型 |
| `environment` / `scale` | `spec.topology` | 推荐拓扑 |
| `businessCriticality` | `spec.topology` | 影响是否高可用 |

### 页面展示建议

只需要显示：

- 中间件类型
- 推荐拓扑
- 用途
- 是否标准套餐

不必在发起页一次性暴露所有底层参数。

---

## 九、外网访问与网络建议映射

这类推荐不一定都应进入 `resources`，要区分“交付资源”和“附属建议”。

## 9.1 可进入 `resources` 的情况

如果明确推荐负载均衡 / 网关，可生成：

```ts
{
  type: 'network',
  name: '订单中心公网访问入口',
  spec: {
    protocol: 'HTTPS',
    port: '443',
    domain: 'xxx.example.com'
  },
  purpose: '承载公网访问流量'
}
```

## 9.2 只作为建议，不直接进 `resources` 的情况

下面内容更适合作为 `adviceNotes`：

- 需要申请证书
- 需要网络策略单
- 需要专项安全评审

因为这些更接近配套流程，不是独立资源本体。

---

## 十、平台接入建议映射

平台接入建议默认不建议直接进入 `resources`。

### 10.1 建议保留在 `adviceNotes` 或附属建议结构

例如：

- 建议开通流水线权限
- 建议开通 EFK 查看权限
- 建议接入统一监控
- 建议接入统一日志平台

### 10.2 如后续需要结构化

可考虑新增：

```ts
platformAccessRecommendations?: [
  { key: 'pipeline', required: true, reason: '容器应用默认建议接入流水线' },
  { key: 'efk', required: true, reason: '需要查看日志' }
]
```

但这不建议在第一轮和 `resources` 混在一起。

---

## 十一、缺失项与非标结果映射

## 11.1 缺失项

建议写入：

```ts
orchestratedPlan.missingInputs = [
  '缺少申请环境，无法确认推荐规格',
  '缺少架构图，无法判断组件依赖'
]
```

### 页面展示

- 发起页：显式提示“待补充”
- 评审页：显示为“推荐依据不足”

## 11.2 非标结果

建议写入：

```ts
orchestratedPlan.recommendationAssessment = {
  standard: false,
  reasonCodes: ['complex_network', 'strict_dr'],
  reasons: ['网络关系复杂，超出标准套餐边界', '容灾要求较高，建议专项评审']
}
```

### 页面展示

- 发起页：显示“非标申请”提示，不改状态
- 评审页：作为重点关注项

---

## 十二、哪些字段只展示、不落资源对象

为避免 `resources` 结构膨胀，建议下面字段只展示或保留在其他结构，不直接落 `resources`：

| 字段 | 建议去向 |
|---|---|
| `requiresPipelineAccess` | 平台接入建议 |
| `requiresEfkAccess` | 平台接入建议 |
| `requiresMonitoring` | 平台接入建议 |
| `requiresLogging` | 平台接入建议 |
| `rtoTarget` | SLA 摘要 / 评审输入 |
| `rpoTarget` | SLA 摘要 / 评审输入 |
| `retentionPeriod` | SLA 摘要 / 评审输入 |
| `restoreRequirement` | SLA 摘要 / 评审输入 |
| `networkPolicyTicketNo` | 评审/流程补充 |
| `reviewRisks` | 评审补充 |

---

## 十三、哪些字段应进入导出但不一定进入页面主展示

这类字段后续可用于导出、评审材料，但不一定需要在发起页主区域显式占位：

| 字段 | 用途 |
|---|---|
| `charset` | 数据库/运维补充 |
| `logRetentionDays` | 运维补充 |
| `resourceRemark` | 补充说明 |
| `diagramIncludes*` | 附件说明 |
| `integrationSystems` | 评审材料 |

---

## 十四、和当前代码结构的衔接建议

建议后续实现时遵循：

1. 页面采集字段来自 `order.answers`
2. 推荐结果落 `orchestratedPlan.resources`
3. 非标和缺失项落 `orchestratedPlan.recommendationAssessment` / `missingInputs` / `adviceNotes`
4. 评审修正落评审补充结构
5. 最终交付结果继续落 `serviceProgress.deliveryDetail`

这样能够避免：

- 直接从页面字段拼交付资产
- 直接从交付结果倒推推荐套餐
- 在 Portal / Center 分别维护两套推荐结果口径

---

## 十五、和主流程状态保护的关系

本草案明确约束：

- `orchestratedPlan.resources` 的变化，不应直接改变主流程状态
- `recommendationAssessment.standard` 不应直接改变主流程状态
- `missingInputs` 只影响提示，不直接回写阶段判断

如果后续确实希望“关键输入缺失时不允许提交”，应仅作为发起页表单校验规则处理，不要直接写进 shared 主流程状态判断链。

---

## 十六、建议下一步

如果继续推进，下一步最合适的是补一份：

`研发落地最小改造清单`

重点回答：

- 第一轮只改哪些类型定义和页面字段
- 哪些字段先不接推荐逻辑，只先收集
- 哪些测试需要补

这样就能从方案文档过渡到真正的实施拆分。
