# IPE/AIOps 现网升级实施清单

## 1. 目标与边界

本次升级目标不是重做云上架构，而是在保留现网访问入口、平台首页与模块切换方式不变的前提下，把 `project-mywayaiops-refactor-V1` 的 Portal / Center 前端替换到现网。

保留项：

- 平台首页：`https://www.getpre.cn/`
- 服务门户：`https://www.getpre.cn/portal/`
- 运营中心：`https://www.getpre.cn/center/`
- 与 `autoops-workspace` 共用平台首页
- 右上角模块切换按钮继续存在

本次升级范围：

- 替换 Portal / Center 静态产物
- 校验与 `autoops-workspace` 的模块入口和切换器兼容
- 保持当前 `/portal/`、`/center/` base 路径
- 如需多人共享演示数据，同步部署 `dev-orders-sync-server.mjs` 并配置 Nginx 反向代理

本次升级暂不做：

- 后端 API 化
- 数据库替换 `localStorage`
- CMDB / ITSM / IaC 真集成，当前先以交付资产台账和 ITSM 审批回填表达
- SSO 改造

## 2. 当前现网形态

根据本次现网实查，`www.getpre.cn` 当前真实形态如下：

- 域名：`www.getpre.cn`
- 平台壳层：`autoops-workspace`
- IPE Portal 静态资源路径：`/portal/`
- IPE Center 静态资源路径：`/center/`
- 容器：`presales-frontend`
- 宿主机真实发布目录：
  - `/home/deploy/project-myway-platform/portal/`
  - `/home/deploy/project-myway-platform/center/`
- 容器内对外映射目录：
  - `/usr/share/nginx/platform/portal/`
  - `/usr/share/nginx/platform/center/`
- Nginx 配置源文件：
  - `/opt/presales-platform/frontend/nginx.conf`

补充说明：

- `/usr/share/nginx/platform` 是宿主机目录 `/home/deploy/project-myway-platform` 的只读挂载。
- 因此现网发布不能只更新 `/usr/share/nginx/html/...`，也不能依赖直接 `docker cp` 到容器内 `platform` 目录。
- 正确方式是覆盖宿主机 `/home/deploy/project-myway-platform/...` 后 reload Nginx。

参考文档：

- [DEPLOY.md](/Users/gjy/project-mywayaiops-refactor-V1/DEPLOY.md:1)
- [与autoops-workspace整合方案.md](/Users/gjy/project-mywayaiops-refactor-V1/docs/与autoops-workspace整合方案.md:1)

## 3. 升级策略

建议采用“原位替换 + 快速回滚”策略。

步骤原则：

1. 不改域名入口。
2. 不改 `portal` / `center` base 路径。
3. 不先动平台壳层主逻辑。
4. 先替换静态资源，再验证模块切换。
5. 每次替换前都保留旧版本备份。

## 5. 标准升级步骤

### 步骤 1：生成待发布版本

```bash
cd /Users/gjy/project-mywayaiops-refactor-V1
./build.sh demo-static
```

生成产物：

- `releases/demo-static/portal/`
- `releases/demo-static/center/`
- `releases/demo-static.tar.gz`

同名版本重复构建时会先清空 `releases/demo-static/`，避免旧静态资源 hash 混入发布包。

### 步骤 2：上传到服务器

```bash
scp /Users/gjy/project-mywayaiops-refactor-V1/releases/demo-static.tar.gz deploy@www.getpre.cn:/home/deploy/
```

### 步骤 3：登录服务器并备份现网

```bash
ssh deploy@www.getpre.cn
cd /home/deploy
```

建议备份当前线上 Portal / Center：

```bash
BACKUP_DIR="backup_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$BACKUP_DIR/portal" "$BACKUP_DIR/center"
cp -r /home/deploy/project-myway-platform/portal/. "$BACKUP_DIR/portal/"
cp -r /home/deploy/project-myway-platform/center/. "$BACKUP_DIR/center/"
```

### 步骤 4：替换静态资源

假设新版本产物已上传到 `/home/deploy/demo-static.tar.gz`：

```bash
rm -rf /home/deploy/demo-static
tar -xzf /home/deploy/demo-static.tar.gz -C /home/deploy
cp -r /home/deploy/demo-static/portal/. /home/deploy/project-myway-platform/portal/
cp -r /home/deploy/demo-static/center/. /home/deploy/project-myway-platform/center/
docker exec presales-frontend nginx -t
docker exec presales-frontend nginx -s reload
```

注意：

- 现网 `/usr/share/nginx/platform` 是只读挂载，不能依赖 `docker cp` 到容器内该目录。
- 如需要排查 Nginx 实际路径，以 `docker exec presales-frontend nginx -T` 为准。
- 线上生效目录是宿主机 `/home/deploy/project-myway-platform/portal/` 和 `/home/deploy/project-myway-platform/center/`。

### 步骤 4.1：可选，部署云上共享数据服务

如果希望不同访问者打开云上页面时能看到同一批演示工单和资产，需要启动共享数据服务。

同步服务默认监听 `0.0.0.0:3011`，便于 `presales-frontend` 容器内 Nginx 通过宿主机网关 `172.18.0.1:3011` 访问。

服务器目录：

```bash
mkdir -p /home/deploy/scripts /home/deploy/.dev-data
```

上传脚本：

```bash
scp /Users/gjy/project-mywayaiops-refactor-V1/src/scripts/dev-orders-sync-server.mjs deploy@www.getpre.cn:/home/deploy/scripts/dev-orders-sync-server.mjs
```

启动服务：

```bash
ssh deploy@www.getpre.cn
cd /home/deploy/scripts
nohup node ./dev-orders-sync-server.mjs > /home/deploy/orders-sync.log 2>&1 &
curl http://127.0.0.1:3011/api/dev/orders-sync
```

预期服务返回：

```json
{"orders":null,"updatedAt":null}
```

`orders:null` 表示服务已启动但尚未写入共享数据，不是启动失败。后续由前端首次同步或手动导入种子数据后会出现订单内容。

日志中如果显示 `listening on http://0.0.0.0:3011/api/dev/orders-sync`，属于正常结果。

如果 `curl` 连接失败，先检查：

```bash
ps -ef | grep dev-orders-sync-server | grep -v grep
cat /home/deploy/orders-sync.log
```

还需要在 Nginx 配置源文件 `/opt/presales-platform/frontend/nginx.conf` 中增加反向代理，再 reload：

```nginx
location /api/dev/orders-sync {
  proxy_pass http://172.18.0.1:3011/api/dev/orders-sync;
  proxy_http_version 1.1;
  proxy_set_header Host $host;
  proxy_set_header X-Real-IP $remote_addr;
  proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  proxy_set_header X-Forwarded-Proto $scheme;
}
```

```bash
docker exec presales-frontend nginx -t
docker exec presales-frontend nginx -s reload
curl -ks https://www.getpre.cn/api/dev/orders-sync
```

## 6. 与 autoops-workspace 的兼容校验

如果现网仍使用 `autoops-workspace` 包装页模式，则本次要重点确认三件事。

### 6.1 模块注册不变

保持：

- 平台首页仍由 `autoops-workspace` 承载
- Portal / Center 仍然是平台模块

### 6.2 模块切换器 URL 不变

应继续指向：

- `/portal/`
- `/center/`

不要改成本地端口或临时测试地址。

### 6.3 包装页和 SPA 共存

若现网仍存在：

- `/modules/ipe-portal/`
- `/modules/ipe-center/`

则应确认它们内部 iframe 或跳转目标仍然是：

- `/portal/`
- `/center/`

不要出现包装页加载旧版本、直接路径加载新版本的双轨不一致。

## 7. 回滚方案

如果出现以下问题，建议立即回滚：

- 首页模块切换失效
- Portal / Center 大面积白屏
- `/portal/` 或 `/center/` 静态资源 404
- 关键流程无法打开

回滚步骤：

```bash
ssh deploy@www.getpre.cn
cd /home/deploy
BACKUP_DIR="backup_20260615_120000"
cp -r "$BACKUP_DIR/portal/." /home/deploy/project-myway-platform/portal/
cp -r "$BACKUP_DIR/center/." /home/deploy/project-myway-platform/center/
docker exec presales-frontend nginx -s reload
```

回滚后再次验证：

- `https://www.getpre.cn/portal/`
- `https://www.getpre.cn/center/`
- 平台首页模块切换

## 8. 升级后立即要补的事

当前 refactor 版虽然可以原位替换到现网，但它本质上仍有大量浏览器本地状态逻辑。云上多人协作时，这会成为下一阶段的主要问题。

建议升级完成后，按以下顺序继续推进：

### 第一优先级：状态后端化

把这些模块从 `localStorage` 迁到 API + DB：

- 工单
- 字段字典
- 模板版本
- 配置档案
- 交付资产

### 第二优先级：账号与权限

补齐：

- 登录态
- 角色权限
- 审计记录

### 第三优先级：外部集成

逐步接入：

- ITSM 审批
- CMDB 正式同步，当前交付资产台账已作为平台内资产沉淀
- 自动交付系统

## 9. 推荐迁移路线

目标：

## 10. 本次建议结论

对于你当前的场景，最稳的路线不是重做一套全新云架构，而是：

1. 先按老版本入口原位升级 Portal / Center
2. 保留 `autoops-workspace` 平台壳与模块切换
3. 升级完成后，再启动 API + DB 改造

这样风险最可控，且最符合当前 `www.getpre.cn` 的现网形态。
