IT基础资源申请辅助填表页面思路规划

本规划面向应用团队申请服务器、对象存储、负载均衡、RabbitMQ 等基础资源时的填表场景。页面以资源清单为起点，把资源规格说明、申请表字段、填写参考和评审关注点联动起来，帮助用户在架构评审前形成更完整、更统一的资源申请材料。

核心定位： 建设一个“资源清单联动的基础资源申请辅助填表工具”。它不替代资源架构评审，不自动做资源选型或开通，只把用户需求梳理成可沟通、可提交、可检查的结构化材料。

一、建设背景与当前痛点

应用团队侧

  • 申请基础资源时需要填写大量表单字段，很多字段并不是业务人员日常熟悉的表达。
  • 不同资源的规格、容量口径、网络参数、安全要求和评审关注点不同，用户容易填错、漏填或描述不一致。
  • 资源清单能告诉用户“有哪些资源”，但没有直接告诉用户“申请这类资源应怎样描述需求”。
  • 低成熟度用户可能只知道业务场景，高成熟度用户希望快速校验已有申请，两类用户都需要更合适的引导方式。

IT团队侧

  • 评审前需要反复确认环境、规模、容量、访问方式、安全要求、高可用要求等基础信息。
  • 同类资源申请材料口径不统一，人工整理和补齐成本较高。
  • 资源标准与申请表缺少联动，标准变更后用户仍可能沿用旧口径填写。
  • 评审人员难以及时区分标准资源申请、需要重点说明的申请、非标准诉求和仍需澄清的内容。
规划主线： 从“用户不知道如何按资源标准填表”出发，将目标收束为“让资源清单直接服务于申请表填写”，最终价值是减少错漏填、减少反复沟通、提升架构评审前材料质量。

二、核心定位与边界

做什么

  • 提供简单 Web 前端或轻量原型。
  • 展示首版资源清单和规格说明。
  • 按资源类型联动申请表字段、填写参考和评审关注点。
  • 提示缺项、待确认项和需要重点说明的内容。
  • 输出评审前沟通和提交所需的结构化材料。

不做什么

  • 不接入 AI 模型。
  • 不联动 ITSM、CMDB、工单系统。
  • 不做审批流转、资源自动开通、自动架构选型。
  • 不替代资源架构评审。
  • 不建设复杂后台、统一研发门户或独立策略引擎。

交付价值

  • 应用团队更容易知道该填什么、怎么填、哪里还不确定。
  • IT团队在评审前拿到更完整的输入材料。
  • 资源清单从静态目录变成填表过程中的参考和约束。
  • 非标准诉求更早暴露，避免混在普通申请中。

三、L0-L5 用户画像分层

L0-L5 用于理解不同用户的填表能力和需要的引导深度，不一定直接作为页面入口暴露给用户。页面可以通过入口、模板、提示、追问和检查能力覆盖这些差异。

层级 用户特征 典型表达 辅助引导重点
L0 代填或完全不确定型 “领导让我申请一套测试环境。” 提供场景模板，允许标记待确认字段，输出需找业务或技术负责人确认的问题清单。
L1 只知道业务场景型 “订单创建后要通知库存和物流。” 从业务动作引导到候选资源，不直接承诺最终资源结论，优先补齐用途、环境、访问对象和规模信息。
L2 技术子域明确型 “我们需要系统间异步解耦。” 把技术能力映射到候选资源类型，帮助用户确认是消息队列、存储、流量入口还是计算资源诉求。
L3 资源明确但规格不清型 “我要一个 RabbitMQ，但不知道选多大。” 展示规格档位、容量口径和填写参考，用少量问题引导选择标准规格或标记为待确认。
L4 架构和规格较清楚型 “生产 RabbitMQ 中型，高可用，预计 5000 msg/s。” 快速生成申请表，只追问缺失字段、冲突信息和评审前必须说明的内容。
L5 检查已有申请或资源清单型 “这份申请是否完整，是否还缺评审信息？” 检查完整性、待确认项、重点关注项和非标准诉求，形成评审前摘要。

四、页面主线

页面主线围绕“资源清单如何帮助用户完成申请表”展开，而不是围绕底层配置或复杂规则展开。用户可以从资源、场景或已有材料进入，但最终都回到申请信息补齐和评审前材料输出。

1. 选择入口

按资源清单、按业务/技术场景、检查已有申请或资源清单。

2. 关联资源

确认资源类型或候选资源，带出标准规格和适用场景。

3. 辅助填表

显示该资源相关字段、填写参考和常见口径。

4. 缺项提示

围绕环境、规模、容量、安全、高可用等信息做补充确认。

5. 输出材料

形成结构化申请信息、缺项清单、待确认项和评审关注点摘要。

五、页面区域构想

资源清单区

  • 展示服务器、对象存储、负载均衡、RabbitMQ 四类首版资源。
  • 每类资源呈现用途、典型场景、规格口径和适用边界。
  • 入口不只是目录浏览，也服务于后续申请表字段联动。

辅助填表区

  • 根据资源类型展示相关申请字段，减少无关字段干扰。
  • 优先使用选项、数字、容量单位和简短说明，减少自由文本歧义。
  • 对不确定信息允许标记为待确认，而不是强行填写。

标准说明 / 评审关注区

  • 说明资源规格、容量口径、填写示例和常见误区。
  • 提示生产环境、公网访问、高可用、大规格、敏感数据等评审关注点。
  • 帮助用户理解为什么某些字段必须补充。

输出预览区

  • 预览结构化申请信息。
  • 单独列出缺项、待确认项和重点说明项。
  • 支持复制或导出为简单文档、表格材料，便于评审前沟通。

六、首版资源范围

服务器 规格档位、CPU/内存/磁盘、操作系统、网络区域、适用负载。 环境、数量、规格、系统盘/数据盘、用途、网络区域、运行系统。 规格依据、生产高可用、安全域、容量增长、是否存在已有资源复用。
对象存储 容量、访问频率、生命周期、权限边界、是否需要公网访问。 容量、年增长量、文件类型、访问方式、保留周期、权限范围。 敏感数据、公网暴露、留存周期、权限控制、容量增长依据。

七、资源清单与申请表联动方式

资源清单不是静态目录。用户选中或确认某类资源后，页面应同步呈现该资源的申请字段、规格说明、填写参考和评审关注点，让“看清单”和“填申请”成为同一个过程。

按资源清单进入

适合 L3-L4 用户。用户已经知道要申请什么资源，页面直接带出该资源的字段、规格和常见评审关注点。

按业务 / 技术场景进入

适合 L0-L2 用户。页面先基于场景给出候选资源和需要确认的问题，再进入对应申请表补齐。

检查已有申请 / 清单

适合 L5 用户。页面重点检查材料完整性、描述口径、待确认项和是否存在非标准诉求。

标准资源 信息基本完整，规格属于清单内标准档位，可形成常规申请材料。
需重点关注 / 附加说明 涉及生产、高可用、公网访问、大规格、敏感数据等，需要在摘要中提示。
非标准诉求 超出标准规格或现有清单能力，页面只整理诉求和理由，不承诺可直接提交。
待澄清 关键字段缺失或描述不明确，输出待确认问题，避免混入正式字段。

八、缺项提示与沟通减少机制

缺项提示的目的不是做复杂判断，而是把 IT 团队高频反复追问的问题前置到填表页面中。首版应优先覆盖影响评审判断的高频信息。

提示类别 典型问题 减少沟通的作用
环境与用途 用于开发、测试、预生产还是生产？承载哪个业务系统？ 避免 IT 团队反复确认资源重要性和评审深度。
规模与容量 预计用户量、并发、吞吐、容量、增长量或峰值是多少？ 让规格选择有依据，减少“先给一个差不多”的模糊申请。
安全与访问 是否公网访问？是否涉及敏感数据？访问权限如何控制？ 提前暴露安全和网络边界问题，便于评审前补充说明。
可用性要求 是否生产高可用？是否需要备份、容灾、保留或恢复要求？ 把高可用和数据保护诉求从资源规格中单独明确出来。
现状与变更 是否已有资源？当前问题是什么？目标变更后的规模是什么？ 区分新申请、扩容、变更和补充材料，减少误判。

九、输出结果

结构化资源申请信息

按资源类型整理环境、用途、规格、容量、访问、安全和高可用等字段，形成统一口径的申请材料。

缺项清单与待确认项

把用户不确定、仍缺失或描述冲突的信息单独列出，便于评审前找对应负责人确认。

评审关注点摘要

汇总生产环境、公网访问、大规格、敏感数据、高可用、非标准诉求等需要在评审中重点说明的内容。

十、首版范围与暂不纳入

首版建议纳入

  • 单页 Web 前端或轻量原型。
  • 四类资源：服务器、对象存储、负载均衡、RabbitMQ。
  • 多入口：按资源清单、按业务/技术场景、检查已有申请或资源清单。
  • 资源清单联动申请字段、规格说明、填写参考和评审关注点。
  • 缺项提示、待确认项、重点关注项、非标准诉求提示。
  • 结构化申请信息预览，以及复制或导出简单材料。

暂不纳入

  • AI 模型理解和自动生成结论。
  • 审批流、工单流转、资源自动开通。
  • ITSM、CMDB、工单系统或资源编排系统联动。
  • 自动容量评估、自动架构选型、成本审批。
  • 复杂后台配置中心、统一研发门户、独立策略引擎。
  • 替代架构、安全、容量、成本等正式评审。

后续支撑能力

字段来源与状态、缺项追问、冲突提示、样例验证、轻量度量可以作为后续支撑能力逐步补充。它们服务于“让用户把申请材料填完整、说清楚”，不应成为首版方案的主线。
